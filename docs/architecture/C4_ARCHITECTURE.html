Level 1 · System Context

Контекст системы

Сервис не заменяет ERP и не пишет в 1C. Он читает ERP-данные из MariaDB, читает 1C через согласованный REST-контракт и сохраняет журнал результатов сверки в ERP-контуре.

Основные пользовательские потоки

Потоки разделены намеренно: матрица взаиморасчетов показывает ERP-сальдовку по поставкам, а документная сверка показывает соответствие ERP и 1C.

Поток A. Матрица взаиморасчетов ERP

Пользователь выбирает ЮЛ, договор, период и лимит
UI не считает всю историю по умолчанию, чтобы не блокировать рабочий контур.
Backend читает поставки и документы ERP
Счета, оплаты, реализации, операции, договоры и поставки берутся из исходных таблиц MariaDB.
Backend считает сальдо по поставкам
Оплаты считаются через get_paidsum, расходы через get_realizsum.
UI показывает долг или переплату
На этом экране не показывается “9 из 9 по 1C”: это не документная сверка.
Пользователь выгружает XLSX
Формат повторяет бухгалтерскую матрицу: спецификация, счета, оплаты, расходы, СФ, плюс/минус.

Поток B. Документная сверка ERP ↔ 1C

Пользователь запускает сверку по поставке или batch
Backend фиксирует scope запуска: client, legal entity, contract, supply/spec или batch.
Backend собирает ERP context
Документы ERP включают code1c, date1c, contract_code1c, amount, VAT, currency.
Backend запрашивает 1C snapshot
Только GET: /reconciliation/v1/snapshot с фильтрами периода, договора, контрагента и поставки.
Matcher сравнивает документы
Primary key: kind + code1c + date1c + contract_code1c. Далее проверяются номер, сумма, НДС, валюта.
Результат сохраняется в журнал
Записываются run summary, item statuses, mismatch fields и комментарии пользователя.

Журнал нужен не для debug, а для управляемого разбора: какие документы расходятся, почему, с какой частотой и какие операции чаще ломают обмен.

Data mapping

Минимальный маппинг для matching

Сверка не должна матчить документы только по коду 1C: в данных есть дубли. Минимальный ключ должен включать тип документа, код 1C, дату 1C и код договора 1C. Остальные поля используются для проверки расхождений.

Поле canonical model ERP 1C REST DTO Роль в сверке
kind Тип ERP-документа: счет, оплата, реализация, операция, СФ/УПД. document_type Часть ключа matching. Разные типы с одинаковым номером не смешиваются.
code1c Код/номер 1C в ERP-поле документа: например kod1c, f_kod1c*, dt1c рядом. code1c или number по договоренному DTO. Часть ключа matching, но не единственное условие.
date1c Дата 1C документа в ERP, проверяется по схеме MariaDB для конкретной таблицы. date Часть ключа и отдельное поле контроля расхождения даты.
contract_code1c Код договора 1C: договор с покупателем / договор с комитентом, например поля семантики CB/CP. contract_code1c Часть ключа matching, защищает от дублей по одинаковому номеру документа.
amount_total Сумма документа ERP в валюте документа. amount_total Проверка AMOUNT_MISMATCH.
vat_rate / vat_amount Ставка/сумма НДС ERP по строке или документу. vat_rate, vat_amount Проверка VAT_MISMATCH. Не подменяется признаком возмещаемости.

Производительность и масштабирование

Live UI должен оставаться быстрым за счет лимитов, фильтров, batch-загрузки и фоновых запусков. Полная историческая сверка не должна выполняться синхронно в браузерном запросе.

< 2 сек
Цель для открытия UI и применения легких фильтров без 1C-запроса.
50–200
Рекомендуемый размер страницы матрицы поставок в live-режиме.
cursor
Пагинация 1C REST API и batch-загрузка документов по списку поставок.
updated_at/hash
Повторная сверка только измененных документов для массового режима.
Нагрузка Решение Что логировать
Пользователь открывает договор с сотнями поставок Пагинация, total count, фильтр проблемных, лимит страницы. Время ERP SQL, количество поставок, количество документов.
Пользователь запускает сверку с 1C по нескольким поставкам Batch endpoint, один 1C snapshot на scope, а не N отдельных запросов. Время 1C REST, размер payload, количество кандидатов matching.
Нужна историческая сверка по всем клиентам Background runner, статус запуска, повтор только по updated_at/hash. run_id, started_at, finished_at, статус, ошибки источников.
Architecture rules

Ключевые правила реализации

Правило Формулировка
Разделение сценариев Матрица взаиморасчетов отвечает за ERP-сальдо по поставкам. Документная сверка отвечает за наличие и совпадение документов ERP ↔ 1C.
1C read-only Сервис вызывает только GET endpoints 1C REST API. Записи в 1C через этот модуль нет.
ERP source of user context Доступ открывается из ERP по short-lived launch token. Логин/пароль ERP не являются production-входом.
Source tables ERP-данные читаются из исходных таблиц и процедур/функций ERP. Зависимость от view_specinv запрещена.
Matching key Минимум: kind + code1c + date1c + contract_code1c. При дублях фиксируется ambiguous/duplicate status.
Business audit Все расхождения пишутся в журнал с типом, полями расхождения и пользовательским комментарием.
