# BABOK требования: сальдовка ERP и сверка ERP vs 1C

## 1. Назначение документа

Документ фиксирует требования к двум связанным, но разным пользовательским сценариям:

1. **Сальдовка взаиморасчетов по поставкам ERP** - бухгалтерский экран и XLSX, который показывает счета, оплаты, возмещаемые расходы, невозмещаемые расходы, долг или переплату по поставкам.
2. **Сверка ERP vs 1C** - контроль полноты и корректности обмена между ERP и 1C по документам, договорам, поставкам, суммам, датам, НДС и аналитике.

Классификация требований дана по BABOK:

- бизнес-требования - зачем нужен модуль и какой бизнес-результат должен быть достигнут;
- требования заинтересованных лиц и пользователей - что должны уметь делать бухгалтер, интегратор, администратор, разработчик 1C и backend-разработчик;
- требования к решению - функциональные и нефункциональные требования;
- переходные требования - что нужно подготовить для внедрения;
- бизнес-правила, ограничения, критерии приемки и трассировка.

## 2. Границы решения

### 2.1 Входит в scope

- Отдельный Python backend сервиса сверки.
- Чтение ERP из MariaDB.
- Чтение 1C через отдельный read-only GET REST API 1C.
- Авторизация пользователя через переход из ERP с короткоживущим launch token.
- Идентификация пользователя через ERP token validation и профиль `veda_users`.
- Экран сальдовки по поставкам ERP.
- XLSX выгрузка сальдовки в формате, привычном бухгалтерии.
- Отдельный экран сверки ERP vs 1C по документам и расхождениям.
- Логирование запусков сверки и типов расхождений.
- Фоновый runner для тяжелых исторических проверок.

### 2.2 Не входит в scope

- Запись или изменение данных в 1C.
- Сырой доступ к OData 1C как контракту сверки.
- Перенос логики в PHP ERP.
- Перенос логики внутрь 1C.
- Использование внешних аналитических таблиц как промышленного источника данных.
- Использование `view_specinv` и других агрегирующих view как источника требований. Для разработки использовать исходные таблицы ERP.

## 3. Бизнес-требования

| ID | Требование | Обоснование | Метрика приемки |
| --- | --- | --- | --- |
| BR-01 | Бухгалтер должен видеть сальдо по каждой поставке клиента. | Сейчас сальдо собирается вручную или через тяжелые таблицы. | Для выбранного клиента/ЮЛ/договора экран показывает список поставок с оплатами, реализацией, долгом или переплатой. |
| BR-02 | Бухгалтер должен выгружать сальдовку в привычный XLSX-формат. | Бухгалтерия работает с формой: `№ спецификации / Счет / Сумма по счету / Сумма оплаты / Возмещаемые / Невозмещаемые / № счф / (+/-)`. | XLSX содержит лист `Акт сверки` с объединенными ячейками и переносами строк в номерах документов. |
| BR-03 | Ответственный за интеграцию должен понимать, совпадают ли ERP и 1C по документам. | Сальдовка ERP сама по себе не доказывает, что 1C содержит те же документы. | Экран сверки ERP vs 1C показывает статусы `ОК`, `Нет в 1C`, `Нет в ERP`, `Сумма расходится`, `Дата расходится`, `НДС расходится`, `Нет аналитики поставки`. |
| BR-04 | Система должна копить историю расхождений. | Нужно анализировать частоту проблем и приоритеты доработок обмена. | Каждый запуск сверки пишет запись в `veda_reconciliation_runs`, каждое расхождение - в `veda_reconciliation_items`. |
| BR-05 | Интеграция с 1C должна быть read-only. | Сервис сверки не должен менять бухгалтерские данные. | Контракт 1C содержит только GET endpoints, сервис не вызывает POST/PUT/PATCH/DELETE в 1C. |
| BR-06 | Тяжелые исторические проверки должны выполняться фоном. | Полная история по всем клиентам может быть дорогой для ERP и 1C. | UI live-запрос ограничен лимитом поставок, полная сверка запускается runner-ом и читается из лога. |

## 5. Функциональные требования

### 5.1 Авторизация и контекст пользователя

| ID | Требование |
| --- | --- |
| FR-AUTH-01 | Пользователь открывает модуль только из ERP. |
| FR-AUTH-02 | ERP передает в модуль короткоживущий `launch_token`, привязанный к пользователю и действию `reconciliation.open`. |
| FR-AUTH-03 | UI отправляет `launch_token` в backend через `POST /api/auth/erp-launch`. |
| FR-AUTH-04 | Backend валидирует `launch_token` через ERP endpoint и получает профиль пользователя. |
| FR-AUTH-05 | Backend ищет пользователя в `veda_users` по `user_id` или login/email, полученным из ERP token validation. |
| FR-AUTH-06 | Backend создает внутреннюю session сервиса сверки и не возвращает `launch_token` в браузер после успешного обмена. |
| FR-AUTH-07 | UI удаляет `launch_token` из адресной строки после обмена через `history.replaceState`. |
| FR-AUTH-08 | В правом верхнем углу UI показывать ФИО пользователя из ERP, если оно найдено. |
| FR-AUTH-09 | Внешняя авторизация вне ERP не используется в production auth-flow. |
| FR-AUTH-10 | Email, ФИО, роли и `user_id`, переданные напрямую из браузера, не считаются доверенными. Доступ разрешается только после успешной проверки token в ERP. |
| FR-AUTH-11 | Сервис сверки не должен читать пароль пользователя из `veda_users` и не должен выполнять вход в ERP по login/password пользователя. |

### 5.2 Сальдовка ERP по поставкам

| ID | Требование |
| --- | --- |
| FR-BAL-01 | Экран сальдовки выбирает поставки по клиенту, ЮЛ, договору, периоду и лимиту. |
| FR-BAL-02 | Иерархия строится как `veda_contacts -> veda_clients -> veda_dogs -> veda_specs`. |
| FR-BAL-03 | Клиент верхнего уровня - `veda_contacts`; ЮЛ - `veda_clients`; договор - `veda_dogs`; поставка/заявка/спецификация - `veda_specs`. |
| FR-BAL-04 | Не использовать `veda_dogs.f_clientid` как надежный источник клиента. Клиента договора определять через `veda_dogs.f_contrid -> veda_clients.f_contactid`. |
| FR-BAL-05 | Тип поставки показывать по справочникам ERP `veda_spr.f_type=33` и `veda_spr.f_type=130`, чтобы различать `Заявка`, `Спецификация` и другие типы. |
| FR-BAL-06 | Сумма оплат берется из ERP routine `get_paidsum(veda_spec_invoices.f_id)`. Для отображения "по счету" сумма связывается со счетом через `veda_schets.f_operid = veda_spec_invoices.f_id`. |
| FR-BAL-07 | Возмещаемые расходы берутся из `get_realizsum` по операциям `veda_spec_invoices.f_isvozm = 1`. |
| FR-BAL-08 | Невозмещаемые расходы берутся из `get_realizsum` по операциям `veda_spec_invoices.f_isvozm = 2` и согласованным операциям, классифицированным ERP как невозмещаемые. |
| FR-BAL-09 | Итог по поставке считать как `оплаты клиента - возмещаемые расходы - невозмещаемые расходы`. |
| FR-BAL-10 | Если итог меньше нуля, показывать `Долг`; если больше нуля, показывать `Переплата`; если равен нулю с учетом допуска копеек, показывать `ОК`. |
| FR-BAL-11 | На экране сальдовки не показывать `9 из 9 ОК по 1C`, потому что это другой сценарий. |
| FR-BAL-12 | Live-экран не должен строить всю историческую сальдовку без лимита. Для больших объемов использовать runner. |

### 5.4 XLSX сальдовки

| ID | Требование |
| --- | --- |
| FR-XLSX-01 | XLSX содержит лист `Акт сверки` в бухгалтерском формате. |
| FR-XLSX-02 | Колонки: `№ спецификации`, пустой разделитель, `Счет`, `Сумма по счету`, `Сумма оплаты`, пустой разделитель, `Возмещаемые расходы`, `Невозмещаемые расходы`, пустой разделитель, `№ счф`, `(+/-)`. |
| FR-XLSX-03 | Ячейки `№ спецификации`, `Возмещаемые расходы`, `Невозмещаемые расходы`, `№ счф`, `(+/-)` объединяются на количество строк счетов поставки. `Сумма оплаты` не объединяется: она выводится построчно по счетам через `get_paidsum(veda_schets.f_operid)`. |
| FR-XLSX-04 | Все номера актов, УПД, счетов-фактур внутри объединенной ячейки `№ счф` выводятся через перенос строки. |
| FR-XLSX-05 | Лист `Правила` содержит понятное описание источника каждого столбца. |
| FR-XLSX-06 | Копирование в TSV должно разворачивать объединенные ячейки в плоские строки с повторением значений. |

### 5.5 Сверка ERP vs 1C

| ID | Требование |
| --- | --- |
| FR-CMP-01 | Сверка ERP vs 1C реализуется как отдельный экран или отдельный режим, не смешанный с сальдовкой. |
| FR-CMP-02 | 1C вызывается только через GET REST API `/reconciliation/v1/...`. |
| FR-CMP-03 | Основной live-запрос к 1C: `GET /reconciliation/v1/snapshot` с параметрами фильтра. |
| FR-CMP-04 | Параметры snapshot: `from`, `to`, `organization_id`, `counterparty_id`, `contract_id`, `contract_number`, `spec_id`, `spec_number`, `document_number`, `document_type`, `include`, `limit`, `cursor`. |
| FR-CMP-05 | Для batch/runner использовать collection endpoints: `counterparties`, `contracts`, `invoices`, `payments`, `sales`, `purchases`, `document-lines`, `account-movements`, `balances`. |
| FR-CMP-06 | ERP и 1C перед сравнением приводятся к canonical model: `counterparty`, `contract`, `customer_invoice`, `incoming_payment`, `sale_act`, `purchase_act`, `document_line`, `account_movement`. |
| FR-CMP-07 | Документы нельзя считать совпавшими только по сумме. |
| FR-CMP-08 | Основной ключ поиска пары документа: `document_kind + code1c + dt1c/date`. Договор 1С используется для сужения кандидатов и затем проверяется как контрольный реквизит. Если `code1c` отсутствует, допускается резервный ключ `document_kind + normalized_number + date + amount + currency`. |
| FR-CMP-09 | Если документ есть в ERP и нет в 1C, статус `Нет в 1C`. |
| FR-CMP-10 | Если документ есть в 1C и нет в ERP, статус `Нет в ERP`. |
| FR-CMP-11 | Если документ найден с отличиями в сумме, дате, номере, договоре или НДС, фиксировать конкретные поля расхождения. Если найден один документ с тем же типом, кодом и датой, но другим договором, показывать `Договор расходится`, а не две ошибки `Нет в 1C` / `Нет в ERP`. |
| FR-CMP-12 | Если источник 1C недоступен, показывать `Ошибка источника 1C`, а не создавать ложные расхождения. |
| FR-CMP-13 | Для строк 1C `Услуги`, где 1C не отдает договор/заявку/аналитику поставки, ставить статус ограничения `Нет аналитики поставки`, а не `ОК`. |
| FR-CMP-14 | Если найдено несколько кандидатов 1C по одному ключу, не выбирать произвольный документ: ставить `Дубли 1C` или `Неоднозначно`. |

### 5.6 Логирование и аналитика расхождений

| ID | Требование |
| --- | --- |
| FR-LOG-01 | Каждый запуск сверки сохранять в `veda_reconciliation_runs`. |
| FR-LOG-02 | Каждое найденное расхождение сохранять в `veda_reconciliation_items`. |
| FR-LOG-03 | В run сохранять scope, параметры, пользователя, источник 1C, время запуска, статус, summary. |
| FR-LOG-04 | В item сохранять тип документа, ключи ERP/1C, статус, сумму ERP, сумму 1C, поля расхождения, `primary_reason`, `severity`, `match_confidence`, `details_json`. |
| FR-LOG-05 | Для аналитики показывать частотность проблем по типу операции, виду документа, контрагенту, договору, периоду. |
| FR-LOG-06 | Пользователь может указать причину разбора и комментарий к строке сверки; комментарий сохраняется в `veda_reconciliation_comments` с привязкой к поставке, run и пользователю. |

## 6. Маппинг данных ERP

### 6.1 Иерархия

| Бизнес-поле | ERP источник | Правило |
| --- | --- | --- |
| Клиент | `veda_contacts` | Верхний клиентский профиль. |
| ЮЛ | `veda_clients` | Юридическое лицо клиента. Связь с клиентом: `veda_clients.f_contactid`. |
| Договор | `veda_dogs` | Договор на ЮЛ. Связь: `veda_dogs.f_contrid -> veda_clients.f_id`. |
| Поставка | `veda_specs` | Поставка/заявка/спецификация. Связь: `veda_specs.f_dogid -> veda_dogs.f_id`. |
| Тип поставки | `veda_spr` | `f_type=33` и `f_type=130`; выводить бизнес-название, не только номер. |

Базовый SELECT для списка поставок:

```sql
SELECT
    s.f_id AS spec_id,
    s.f_num AS spec_num,
    s.f_dt AS spec_date,
    spec_type.f_name AS spec_type,
    spec_subtype.f_name AS spec_subtype,
    d.f_id AS dog_id,
    d.f_dogname AS dog_number,
    d.f_kod1c AS dog_code1c,
    cl.f_id AS legal_id,
    cl.f_cname AS legal_name,
    cl.f_inn AS legal_inn,
    contact.f_id AS contact_id,
    COALESCE(contact.f_cname, contact.f_name, cl.f_cname) AS contact_name
FROM veda_specs s
JOIN veda_dogs d ON d.f_id = s.f_dogid
JOIN veda_clients cl ON cl.f_id = d.f_contrid
LEFT JOIN veda_contacts contact ON contact.f_id = cl.f_contactid
LEFT JOIN veda_spr spec_type ON spec_type.f_type = 33 AND spec_type.f_num = s.f_typez
LEFT JOIN veda_spr spec_subtype ON spec_subtype.f_type = 130 AND spec_subtype.f_num = s.f_subtype
WHERE
    (:scope = 'legal' AND cl.f_id = :client_id)
    OR (:scope = 'contact' AND (cl.f_contactid = :client_id OR contact.f_id = :client_id))
ORDER BY s.f_dt DESC, s.f_id DESC
LIMIT :limit;
```

### 6.2 Суммы сальдовки

| Колонка | ERP источник | Правило |
| --- | --- | --- |
| `Счет` | `veda_schets` | Только счета покупателю: `veda_schets.f_dogtype=2`, `veda_schets.f_dogid=spec_id`, `veda_schets.f_type=1`. `f_type=2` относится к счету от поставщика. |
| `Сумма по счету` | `veda_schets.f_sum` | Сумма счета покупателю. Если валюты разные, не суммировать без утвержденного курса. |
| `Сумма оплаты` | `get_paidsum(veda_spec_invoices.f_id)` + связь `veda_schets.f_operid` | Сумма оплат клиента по операции счета. В итог поставки идет сумма `get_paidsum` по операциям поставки; в XLSX и детализации счета сумма показывается напротив счета, если `veda_schets.f_operid` заполнен. |
| `Возмещаемые расходы` | `get_realizsum(veda_spec_invoices.f_id)` | Сумма реализации по операциям `f_isvozm=1`. |
| `Невозмещаемые расходы` | `get_realizsum(veda_spec_invoices.f_id)` | Сумма реализации по операциям `f_isvozm=2` и согласованной классификации ERP. |
| `№ счф` | ERP закрывающие документы и 1C документы при сверке | Все номера документов выводятся через перенос строки. |
| `(+/-)` | Расчетное поле | `Сумма оплаты - Возмещаемые расходы - Невозмещаемые расходы`. |

Базовый SELECT для оплат:

```sql
SELECT
    h.f_id AS payment_id,
    h.f_kod1C AS code1c,
    d.f_docid AS oper_id,
    h.f_ppnum AS payment_number,
    COALESCE(NULLIF(h.f_dt1C, '0000-00-00 00:00:00'), h.f_ppdt) AS payment_date,
    h.f_sum AS payment_sum,
    d.f_clssum AS classified_sum,
    h.f_type AS payment_type,
    oper.f_isvozm AS reimbursement_id
FROM veda_acchist_docs d
JOIN veda_acchist h ON h.f_id = d.f_acchistid
JOIN veda_spec_invoices oper ON oper.f_id = d.f_docid
WHERE d.f_doctype = 3
  AND oper.f_id IN (:operation_ids);
```

Базовый SELECT для операций:

```sql
SELECT
    oper.f_id AS oper_id,
    oper.f_num_oper AS oper_number,
    oper.f_sum AS oper_sum,
    oper.f_isvozm AS reimbursement_id,
    oper.f_nds AS vat_id,
    oper.f_invoiceid AS invoice_id,
    get_paidsum(oper.f_id) AS paid_sum,
    get_realizsum(oper.f_id) AS realiz_sum,
    get_expensessum(oper.f_id) AS expenses_sum,
    get_profit(oper.f_id) AS profit_sum
FROM veda_spec_invoices oper
LEFT JOIN veda_categs oper4_specs
    ON oper4_specs.f_objectid = oper.f_id
   AND oper4_specs.f_ctgtype = 24
   AND oper4_specs.f_objecttype = 5
WHERE oper.f_parenttype IN (2, 4)
  AND (
      CASE oper.f_parenttype
          WHEN 2 THEN oper.f_specid
          WHEN 4 THEN CAST(oper4_specs.f_valstr AS SIGNED)
          ELSE NULL
      END
  ) = :spec_id;
```

## 7. Маппинг данных 1C REST

1C должна отдавать DTO по OpenAPI-контракту:

- `contracts/onec/openapi.json`
- `contracts/onec/openapi.yaml`
- `contracts/onec/openapi_ru.md`

Основной запрос:

```http
GET /reconciliation/v1/snapshot?from=2026-06-01&to=2026-06-30&organization_id=...&counterparty_id=...&contract_number=...&spec_number=...&include=invoices,payments,sales,purchases,document_lines,balances&limit=100&cursor=...
```

Тело запроса отсутствует. Все фильтры передаются query-параметрами. Ответ JSON.

Обязательные поля DTO для сравнения:

| Поле DTO | Назначение |
| --- | --- |
| `source_system` | Всегда `1c`. |
| `source_id` | Уникальный идентификатор 1C объекта. |
| `document_type` | Нормализованный тип: `customer_invoice`, `incoming_payment`, `sale_act`, `purchase_act`, `document_line`, `account_movement`. |
| `number` | Номер документа 1C. |
| `date` | Дата документа. |
| `organization_id` | Код/ID организации. |
| `counterparty_id` | Код/ID контрагента. |
| `contract_id` | Код/ID договора 1C. |
| `contract_number` | Номер договора. |
| `spec_id` / `spec_number` | Код или номер поставки/заявки/спецификации, если доступен. |
| `amount_total` | Сумма документа или строки. |
| `vat_rate`, `vat_amount` | Ставка и сумма НДС. |
| `currency` | Валюта. |
| `posted` | Признак проведения. |
| `deleted` | Признак удаления. |
| `updated_at` | Дата изменения. |
| `hash` | Хэш значимых полей для ускорения повторной сверки. |

Ключ сопоставления документов:

1. Основной ключ поиска пары: `document_type + 1C document code + date`.
2. Если по этому ключу найдено несколько кандидатов, сначала сужать кандидатов по `contract_id` / коду договора 1C.
3. Если найден один документ с тем же типом, кодом и датой, но другим договором, фиксировать `CONTRACT_MISMATCH`.
4. Если договор в ответе 1C отсутствует, но документ найден, фиксировать `CONTRACT_CONTEXT_MISSING`, когда договор нужен для привязки к поставке.
5. Если код 1C отсутствует в ERP, используется резервный ключ `document_type + normalized(number) + date + amount_total + currency`.
6. Сопоставление только по коду 1C запрещено: в ERP и 1C встречаются дубли кодов за разные даты/договоры.

## 8. Статусы сверки ERP vs 1C

| Статус | Когда ставится | Что показывать пользователю |
| --- | --- | --- |
| `MATCH` | Документ найден в обеих системах, ключевые поля совпали. | `ОК`. |
| `NOT_FOUND_IN_1C` | В ERP заполнен код 1C, но документ не найден в 1C по типу, коду и дате; точный fallback также пуст. | `Не найдено в 1C`. |
| `ERP_CODE1C_MISSING` | ERP-документ существует, но код 1C не заполнен, поэтому прямую проверку выполнить нельзя. | `Нет кода 1C в ERP`. |
| `NOT_FOUND_IN_ERP` | Документ 1C не найден в исходных ERP-таблицах глобальным поиском по типу, коду 1C и дате. | `Не найдено в ERP`. |
| `NOT_LINKED_TO_DELIVERY_IN_ERP` | Документ найден в ERP глобально, но не связан с выбранной поставкой. | `Нет связи ERP`. |
| `AMOUNT_MISMATCH` | Документ найден, но сумма расходится. | `Сумма расходится`. |
| `DATE_MISMATCH` | Документ найден, но дата расходится. | `Дата расходится`. |
| `NUMBER_MISMATCH` | Документ найден, но номер/код расходится. | `Номер расходится`. |
| `CONTRACT_MISMATCH` | Документ найден, но договор не совпал. | `Договор расходится`. |
| `VAT_MISMATCH` | Отличается ставка или сумма НДС. | `НДС расходится`. |
| `DUPLICATE_IN_1C` | По ключу `код 1C + дата + договор` найдено несколько документов 1C. | `Дубли в 1C`. |
| `AMBIGUOUS_MATCH` | Резервный ключ вернул несколько кандидатов, однозначное сопоставление невозможно. | `Неоднозначно`. |
| `CONTRACT_CONTEXT_MISSING` | Для строки 1C нет договора/поставки/аналитики. | `Нет аналитики поставки`. |
| `SOURCE_ERROR` | ERP или 1C недоступны либо вернули ошибку. | `Ошибка источника`. |

## 11. Бизнес-правила и ограничения

1. Сальдовка ERP не является доказательством совпадения ERP и 1C. Это отдельный бухгалтерский инструмент.
2. Сверка ERP vs 1C должна показывать документы и расхождения, а не только итоговое сальдо.
3. `f_isvozm=1` трактуется как возмещаемые расходы.
4. `f_isvozm=2` трактуется как невозмещаемые расходы, если ERP-логика операции не классифицирует ее иначе.
5. Тип счета покупателю: `veda_schets.f_type=1`. Тип счета поставщика: `veda_schets.f_type=2`. Перед rollout это значение нужно подтвердить на production-справочнике ERP.
6. Для строк 1C `Услуги` договор с клиентом нельзя подставлять искусственно, если это расходы подрядчиков и в 1C нет аналитики поставки. Такой кейс фиксируется как ограничение аналитики.
7. Письма и внешние источники, которых нет в ERP и 1C, не участвуют в автоматическом расчете.
8. Корректировки поставок учитывать как часть истории поставки, но не смешивать основной отчет и корректировки без явного признака/типа.
9. Ошибка источника не является расхождением данных.
10. Пользовательский UI не должен показывать технические термины инфраструктуры.

## 12. Критерии приемки

| ID | Критерий |
| --- | --- |
| AC-01 | При выборе клиента и договора экран сальдовки показывает несколько поставок с агрегатами и детализацией. |
| AC-02 | На сальдовке нет статусов `1C ОК` и технических источников. Есть только `Долг`, `Переплата`, `ОК`. |
| AC-03 | XLSX выгружается по кнопке и содержит бухгалтерский формат с объединенными ячейками. |
| AC-04 | В XLSX все номера закрывающих документов внутри объединенной ячейки идут через перенос строки. |
| AC-05 | `GET /reconciliation/v1/snapshot` вызывается без тела запроса и возвращает JSON DTO по OpenAPI. |
| AC-06 | Экран сверки ERP vs 1C показывает не менее трех классов исходов: совпало, нет в 1C/ERP, поля расходятся. |
| AC-07 | При недоступной 1C пользователь видит `Ошибка источника 1C`, а лог run получает статус ошибки источника. |
| AC-08 | После запуска сверки в БД есть run и item-записи с типами расхождений. |
| AC-09 | Для тяжелой сверки по периоду UI запускает background runner, а не блокирует live-запрос. |
| AC-10 | После загрузки нескольких поставок UI показывает нейтральный результат загрузки, а отдельная подсказка просит отметить кружок в колонке `Выбор`; sticky-панель выбранной поставки скрыта до явного выбора. |
| AC-11 | Попытка открыть `Сверка ERP/1C` без выбранной поставки не переводит пользователя в пустой экран, а подсвечивает матрицу выбора. |
| AC-12 | После выбора поставки появляется sticky-панель с клиентом, договором, кодами договоров 1C, сальдо и действиями `Сверить с 1C`, `Показать документы`, `Скачать поставку XLSX`. |
| AC-13 | После выполнения сверки кнопка повторного запуска называется `Обновить сверку`, а экспорт результата доступен как `Скачать сверку XLSX`. |
| AC-14 | Экспорт списка без полного серверного batch-экспорта называется `Скачать видимые строки XLSX`; пользователь не ожидает выгрузку всех найденных поставок за пределами текущей страницы. |
| AC-15 | Если UI-фильтр скрывает выбранную поставку, выбранный контекст сбрасывается и кнопки сверки/экспорта поставки становятся недоступны до нового выбора. |
