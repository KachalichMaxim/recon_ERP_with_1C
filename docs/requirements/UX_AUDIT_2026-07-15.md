# UX-аудит интерфейса сверки от 15.07.2026

## Сценарий пользователя

1. Бухгалтер открывает сервис из ERP и попадает в матрицу взаиморасчетов.
2. Выбирает ЮЛ и договор либо сразу находит известную поставку.
3. Ограничивает период и загружает страницу поставок.
4. Просматривает агрегаты по клиенту, ЮЛ и договору, затем раскрывает нужную поставку.
5. При необходимости переходит к документной сверке ERP и 1С или выгружает XLSX.

## Выявленные затруднения и выполненные изменения

| Затруднение | Риск | Изменение |
| --- | --- | --- |
| Известную поставку можно было найти только просмотром страниц матрицы. | Долгий поиск и высокая вероятность выбрать не ту строку. | Добавлен autocomplete по полному ERP-названию и точному `spec_id`. Текстовый поиск начинается с трех символов. |
| Верхние уровни `Клиент` и `ЮЛ` фактически повторяли одну запись. | Пользователь не понимает иерархию группы компаний. | Верхний уровень строится по `veda_contacts`, уровень ЮЛ - по `veda_clients`. |
| Поле `Поставок` выглядело как бизнес-фильтр количества поставок. | Неясно, ограничивает ли поле данные или страницу. | Поле переименовано в `Строк на странице`. |
| После смены клиента могли оставаться договор и поставка старого клиента. | Пустой результат без понятной причины. | Смена клиента очищает договор и поставку; смена договора очищает поставку. |
| Агрегатные строки раскрывались только мышью. | Снижение доступности и скорости работы с клавиатуры. | Добавлены фокус, `Enter` и `Space`, а также `aria-expanded`. |
| Технический ID мог становиться главным обозначением поставки. | Бухгалтеру сложно идентифицировать объект. | В подсказке главным является полное название; ID показан вторичным текстом. |

## Источник названия поставки

Production-запрос не использует `view_specs`. Название воспроизводит его бизнес-формат из исходных таблиц:

```text
veda_dogs.f_dogname / veda_specs.f_num / организация veda_clients.f_abbr / ЮЛ veda_clients.f_cname
```

Контрольный пример: `660/1/1051/ВА/АЭРО-ТРЕЙД`, `spec_id=20334`.

## Контроль производительности

На production MariaDB 15.07.2026:

| Проверка | Результат |
| --- | ---: |
| Точный поиск `spec_id=20334` | 0,34-0,77 с |
| Поиск `660/1/1051` | 0,40-0,79 с |
| Загрузка матрицы одной найденной поставки с итогами | 3,72 с, ERP SQL 3,33 с |

## Оставшиеся риски

- Статусный фильтр применяется к загруженной странице, а не ко всей истории договора; интерфейс должен явно сохранять это различие.
- Для массовой истории остаются обязательными пагинация и background runner.
- Визуальная проверка пиксельного соответствия, контраста и мобильных состояний в этом цикле не выполнялась; аудит проведен по коду, структуре данных и рабочим API по согласованию с заказчиком.
