# Границы MVP

## Цель

MVP помогает пользователю провести первичный разбор полноты передачи документов из ERP в 1С по выбранной поставке и связанным с ней операциям.

Поставка является точкой входа пользователя, но не строгой бухгалтерской границей. Сервис собирает доступный граф связанных документов, строк и распределений, в том числе проведенных в 1С по другим договорам.

## Что делает MVP

- строит ERP-сальдовку по выбранным поставкам;
- получает ERP-контекст выбранной поставки;
- проверяет известные ERP-документы в доступной области 1С;
- показывает дополнительные документы 1С, возвращенные поставочным scope;
- диагностирует отсутствие ERP-счета, закрывающего документа или кода 1С;
- разделяет отсутствие документа и отсутствие доказанной связи с поставкой;
- показывает расхождения суммы, даты, номера, валюты и НДС;
- сохраняет запуск, причины и комментарии разбора;
- дает ссылки на исходный документ и операцию ERP;
- выгружает сальдовку и результат проверки в XLSX.

## Что MVP не гарантирует

- полную сверку всех документов организации, клиента или договора;
- полноту 1С, если документ потерял все связи с поставкой;
- независимое сравнение двух полных populations ERP и 1С;
- автоматическое определение системы, в которой данные являются правильными;
- окончательное бухгалтерское подтверждение закрытия поставки;
- утвержденность экспериментальных бизнес-правил без проверки пользователями.

## Интерпретация результата

- `execution_status=completed`: техническое выполнение завершено;
- `coverage_status=delivery_snapshot`: проверена доступная поставочная область 1С;
- `coverage_status=delivery_snapshot_with_warnings`: 1С вернула предупреждения;
- `result_status=issues_found`: в доступной области есть строки для разбора;
- `result_status=no_issues_in_available_scope`: в доступной области проблем не найдено, но независимая полнота 1С не доказана;
- `balance_status`: отдельный результат сравнения сальдо, не заменяющий документную проверку.

## Критерии полезности

MVP считается полезным, если пользователь без ручного поиска по нескольким системам:

1. называет основную причину проблемы;
2. определяет сторону и владельца исправления;
3. переходит к исходному объекту;
4. понимает, как проверить устранение проблемы.
