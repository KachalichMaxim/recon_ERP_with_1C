# ТЗ: маппинг ERP и 1С для встроенного экрана акта сверки

## 1. Цель

Разработать экран сверки взаиморасчетов, который сравнивает данные ERP с данными сервиса 1С.

Документ описывает не инфраструктуру, а требования к данным:

- что брать из ERP;
- что запрашивать в 1С;
- какие поля сравнивать;
- как строить иерархию клиент / ЮЛ / договор / поставка;
- как классифицировать счета, оплаты, возмещаемые и невозмещаемые расходы;
- какие статусы показывать разработчику и бухгалтеру.

Целевой backend сверки - Python-сервис. ERP читается из MariaDB, 1С читается через согласованный read-only REST API.

## 2. Основной принцип сверки

ERP является источником управленческой структуры:

```text
Клиент -> ЮЛ -> договор -> поставка/заявка/спецификация
```

1С является источником бухгалтерских документов:

```text
контрагенты, договоры, счета, банковские документы, акты, УПД, акты сверки
```

Сверка должна отвечать на вопрос:

```text
какой ERP-документ или ERP-распределение соответствует какому документу/строке 1С
```

Сумма сама по себе не является ключом поиска. Сумма только подтверждает уже найденную пару.

Базовый ключ поиска пары документа:

```text
тип документа + код 1С + дата 1С
```

Код договора 1С не является единственным обязательным ключом поиска. Он используется:

- чтобы сузить кандидатов, если по типу, коду и дате найдено несколько документов;
- чтобы подтвердить, что документ привязан к нужной поставке;
- чтобы поставить расхождение `Договор`, если документ найден, но договор отличается;
- чтобы поставить ограничение `Нет аналитики поставки`, если документ найден, но договор/аналитика поставки не переданы 1С.

Для строк и распределений дополнительно используются:

- договор 1С;
- договор-заявка / спецификация 1С;
- счет на оплату;
- строка табличной части;
- сумма строки;
- тип строки 1С.

Код договора 1С берется из ERP как обязательный контрольный реквизит:

- договор с покупателем / клиентский договор поставки: `veda_specs.f_kod1cb`;
- договор с комитентом: `veda_specs.f_kod1cp`;
- базовый договор: `veda_dogs.f_kod1c`.

Если в 1С найден один документ с тем же типом, кодом и датой, но по другому договору, это не `Нет в 1С` / `Нет в ERP`, а расхождение `Договор`. Если кандидатов несколько и договор не позволяет выбрать один документ, статус `Неоднозначно`.

## 3. ERP-иерархия клиента

Для клиентской иерархии использовать только цепочку:

```text
veda_contacts -> veda_clients -> veda_dogs -> veda_specs
```

Маппинг:

| Уровень | ERP | Ключевая связь | Назначение |
|---|---|---|---|
| Клиент | `veda_contacts` | `veda_clients.f_contactid` | верхний уровень клиента |
| ЮЛ клиента | `veda_clients` | `veda_dogs.f_contrid = veda_clients.f_id` | юридическое лицо клиента |
| Договор | `veda_dogs` | `veda_specs.f_dogid = veda_dogs.f_id` | базовый договор с ЮЛ |
| Поставка | `veda_specs` | `veda_specs.f_dogid` | заявка / спецификация / поставка |

Нельзя строить иерархию через старые альтернативные связи, если они противоречат `veda_dogs.f_contrid -> veda_clients.f_id`.

Базовый запрос контекста поставки:

```sql
SELECT
    contact.f_id      AS contact_id,
    contact.f_name    AS contact_name,
    legal.f_id        AS legal_id,
    legal.f_cname     AS legal_name,
    legal.f_inn       AS legal_inn,
    legal.f_kpp       AS legal_kpp,
    dog.f_id          AS dog_id,
    dog.f_dogname     AS dog_name,
    dog.f_dogdate     AS dog_date,
    spec.f_id         AS spec_id,
    spec.f_num        AS spec_number,
    spec.f_dt         AS spec_date
FROM veda_specs spec
JOIN veda_dogs dog
    ON dog.f_id = spec.f_dogid
JOIN veda_clients legal
    ON legal.f_id = dog.f_contrid
LEFT JOIN veda_contacts contact
    ON contact.f_id = legal.f_contactid
WHERE spec.f_id = :spec_id;
```

## 4. 1С-сервис, который учитывать

Для экрана сверки использовать только чтение из 1С через read-only REST API. Контракт REST API описан в `docs/api/onec_reconciliation_rest_api_tz.md`, OpenAPI - в `contracts/onec`.

1С должна отдавать нормализованные DTO по документам, справочникам, строкам документов, движениям и техническим идентификаторам. Python backend не должен вызывать write-side методы 1С и не должен зависеть от внутренних OData-имен документов 1С.

## 5. Маппинг справочников

### 5.1 Клиент / ЮЛ ERP -> Контрагент 1С

ERP:

- `veda_contacts` - клиент верхнего уровня;
- `veda_clients` - ЮЛ клиента;
- `veda_clients.f_inn`;
- `veda_clients.f_kpp`;
- код 1С ЮЛ хранится в текущих ERP-полях/таблицах обмена, используемых `c1c_createClient`.

1С:

- `Справочник.Контрагенты`;
- функция `getClients(inn, name, grp, contrid, stat)`.

Поля 1С из ответа `getClients`:

- `Код`;
- `Наименование`;
- `ПолноеНаименование`;
- `ИНН`;
- `КПП`;
- `ЮридическоеФизическоеЛицо`;
- `ОП`;
- `ГО`;
- `ГК`;
- `Удален`;
- `dogs[]`;
- `accs[]`.

Сравнивать:

| ERP | 1С | Правило |
|---|---|---|
| код 1С ЮЛ | `Код` | основной ключ контрагента |
| `veda_clients.f_inn` | `ИНН` | контрольное поле |
| `veda_clients.f_kpp` | `КПП` | контрольное поле |
| наименование ЮЛ | `Наименование` / `ПолноеНаименование` | предупреждение при отличии |
| признак активности | `Удален` | удаленный контрагент не считается полноценным совпадением |

Если код 1С отсутствует в ERP, допускается поиск по ИНН/КПП только для диагностики. Такой результат не должен автоматически считаться надежной связью.

### 5.2 Организация ERP -> Организация 1С

ERP:

- организация сделки/поставки из ERP;
- код 1С организации, который используется при вызовах `createDog`, `createSpec`, `createInvoice`, `addupdAkt`, `addupdAktR`, `updAccHist`.

1С:

- `Справочник.Организации`;
- функция `getOrgs()`.

Поля 1С:

- `kod1c`;
- `name`;
- `inn`.

Сравнивать:

| ERP | 1С | Правило |
|---|---|---|
| код 1С организации | `kod1c` | основной ключ организации |
| ИНН организации | `inn` | контрольное поле |
| название организации | `name` | предупреждение при отличии |

## 6. Маппинг договоров и поставок

### 6.1 Базовый договор ERP -> Договор контрагента 1С

ERP:

- `veda_dogs.f_id`;
- `veda_dogs.f_dogname`;
- `veda_dogs.f_dogdate`;
- `veda_dogs.f_contrid`;
- `veda_dogs.f_orgid`;
- `veda_dogs.f_kod1c`;
- `veda_dogs.f_kod1cg`;
- `veda_dogs.f_kod1cgo`;
- `veda_dogs.f_kod1cgc`;
- валюта договора.

1С:

- `Справочник.ДоговорыКонтрагентов`;
- договоры приходят внутри `getClients().dogs[]`;
- создание/обновление описано функцией `createDog`.

Поля 1С из `dogs[]`:

- `dognum`;
- `dogdt`;
- `orgname`;
- `orgid`;
- `dogval`;
- `dogtype`;
- `dogcode`.

Сравнивать:

| ERP | 1С | Правило |
|---|---|---|
| `veda_dogs.f_kod1c` | `dogcode` | основной ключ договора |
| `veda_dogs.f_dogname` | `dognum` | контроль номера |
| `veda_dogs.f_dogdate` | `dogdt` | контроль даты |
| код 1С организации | `orgid` | договор должен быть в той же организации |
| валюта договора | `dogval` | контроль валюты |
| тип договора ERP | `dogtype` | контроль вида договора |

Для клиентского договора `dogtype = 100` функция `createDog` дополнительно создает группы:

- `<dogname>(гр)`;
- `Открытые`;
- `Закрытые`.

Эти группы нужны для структуры спецификаций в 1С, но сами по себе не являются документами сверки.

### 6.2 Поставка ERP -> договор-заявка / договор-спецификация 1С

ERP:

- `veda_specs.f_id`;
- `veda_specs.f_num`;
- `veda_specs.f_dt`;
- `veda_specs.f_dogid`;
- `veda_specs.f_kod1cb`;
- `veda_specs.f_kod1cp`.

1С:

- `Справочник.ДоговорыКонтрагентов`;
- создание описано функцией `createSpec`.

Что создает `createSpec`:

| Возврат 1С | Объект 1С | Когда нужен |
|---|---|---|
| `specgrn` | группа спецификации / заявки | контейнер в группе `Открытые` |
| `specb` | договор с покупателем, вид `СПокупателем` | клиентский взаиморасчет, счета и оплаты |
| `speca` | агентский договор, вид `СКомитентомНаЗакупку` | агентские/комитентские расчеты, создается для `subtype = 2/5/6` |
| `specbd` | параметры счетов 62.01/62.02 | настройка договора покупателя |
| `specad` | параметры счетов 76.06 | настройка агентского договора |

Сравнивать:

| ERP | 1С | Правило |
|---|---|---|
| `veda_specs.f_kod1cb` | `dogcode` договора `СПокупателем` | основной ключ клиентского договора поставки |
| `veda_specs.f_kod1cp` | `dogcode` договора `СКомитентомНаЗакупку` | основной ключ агентского договора поставки |
| `veda_dogs.f_dogname + '/' + veda_specs.f_num` | `dognum` | контроль номера договора-заявки |
| дата поставки/договора | `dogdt` | контроль даты |
| код 1С ЮЛ | владелец договора 1С | договор должен принадлежать тому же контрагенту |
| код 1С организации | организация договора 1С | договор должен быть в той же организации |

Если в 1С-строке документа нет договора-заявки или кода договора, полный match по поставке невозможен.

## 7. Маппинг счетов

### 7.1 Счет ERP -> Счет на оплату покупателю 1С

ERP:

- `veda_schets.f_id`;
- `veda_schets.f_num`;
- `veda_schets.f_dt`;
- `veda_schets.f_sum`;
- `veda_schets.f_val`;
- `veda_schets.f_status`;
- `veda_schets.f_orgid`;
- `veda_schets.f_contrid`;
- `veda_schets.f_dogid`;
- `veda_schets.f_operid`;
- `veda_schets.f_maininv`;
- `veda_schets.f_kod1c`.

Фильтр счетов покупателю в ERP:

```sql
veda_schets.f_dogtype = 2
AND veda_schets.f_dogid = :spec_id
AND COALESCE(veda_schets.f_type, 0) = 1
```

Бизнес-правило: `f_type=1` - счет покупателю; `f_type=2` - счет от поставщика. Перед rollout нужно сверить это значение на production-справочнике ERP.

1С:

- `Документ.СчетНаОплатуПокупателю`;
- создание описано функцией `createInvoice`.

Поля, которые записывает `createInvoice`:

- `Номер`;
- `Дата`;
- `Организация`;
- `Контрагент`;
- `ДоговорКонтрагента`;
- `ВалютаДокумента`;
- `СуммаДокумента`;
- `СтруктурнаяЕдиница`;
- `ПодразделениеОрганизации`;
- строки `Товары`;
- статус оплаты счета.

Сравнивать:

| ERP | 1С | Правило |
|---|---|---|
| `veda_schets.f_kod1c` или `f_num` | `Номер` | ключ счета с учетом даты и организации |
| `veda_schets.f_dt` | `Дата` | дата документа |
| код 1С организации | `Организация.Код` | обязательное совпадение |
| код 1С ЮЛ | `Контрагент.Код` | обязательное совпадение |
| код договора поставки | `ДоговорКонтрагента.Код` | обязательное совпадение для полного match |
| `veda_schets.f_sum` | `СуммаДокумента` | контроль суммы |
| `veda_schets.f_val` | `ВалютаДокумента.Код` | контроль валюты |
| `veda_schets.f_status` | статус оплаты 1С | контроль статуса |

Если счет агрегирует дочерние счета, в матрице надо показывать связь через `f_maininv`, но сверять сумму поставки нужно по фактическим строкам и распределениям.

## 8. Маппинг оплат

### 8.1 Оплата ERP -> Поступление/Списание на расчетный счет 1С

ERP:

- `veda_acchist`;
- `veda_acchist_docs`.

Ключевые ERP-поля:

- `veda_acchist.f_kod1C`;
- `veda_acchist.f_dt1C`;
- `veda_acchist.f_ppnum`;

1С:

- `Документ.ПоступлениеНаРасчетныйСчет`;

- чтение через `getAcchist(dtb, dte, orgid)`;

Поля 1С из `getAcchist`:

- `orgid`;
- `orgacc`;
- `orgbic`;
- `orgaccid`;
- `typd`: `0` - поступление, `1` - списание;
- `dtpp`;
- `numpp`;
- `sumpp`;
- `typeop`;
- `contrid`;
- `contrinn`;
- `contracc`;
- `contrbic`;
- `contraccid`;
- `val`;
- `dognum`;
- `dogid`;
- `vhdt`;
- `vhnum`;
- `grndpp`;
- `dop[]`.

Поля строки расшифровки `dop[]`:

- `contrdog`;
- `sdelka`;
- `sumpl`;
- `kursvz`;
- `sumvz`;
- `invopl`;
- `invstatus`;
- `invoplnum`;
- `invopldt`;
- `sumusl`;
- `kurspr`.

Сравнивать:

| ERP | 1С | Правило |
|---|---|---|
| `veda_acchist.f_kod1C` | `numpp` | код/номер банковского документа 1С; сам по себе не является уникальным ключом |
| `veda_acchist.f_dt1C` | `dtpp` | дата банковского документа 1С; обязательная часть ключа сверки |
| направление оплаты | `typd` | поступление/списание должны совпасть по смыслу |
| код 1С организации | `orgid` | обязательное совпадение |
| код договора | `dogid` или `dop[].contrdog` | контекст договора |
| счет на оплату ERP | `dop[].invoplnum` / `dop[].invopl` | связь оплаты со счетом |
| полная сумма выписки | `sumpp` | контроль документа целиком |
| сумма оплаты операции ERP | `get_paidsum(veda_spec_invoices.f_id)` | использовать для сальдо поставки и построчной оплаты по счету через `veda_schets.f_operid` |
| строка распределения ERP | `veda_acchist_docs.f_clssum` | использовать как аудиторскую расшифровку платежного документа, но не как итоговую сумму оплаты в матрице |

Ключ полного сопоставления оплаты: `тип документа + veda_acchist.f_kod1C + veda_acchist.f_dt1C + dogid`.
Если одна выписка распределена по нескольким поставкам, в матрице использовать результат `get_paidsum(oper_id)` по операциям выбранной поставки. В документной сверке строки `veda_acchist_docs` применяются для расшифровки и контроля связи платежного документа с операцией.
Матч только по `f_kod1C` запрещен из-за дублей.

Для матрицы поставки нельзя брать всю сумму банковского документа, если оплата распределена между несколькими поставками. В строке поставки использовать `SUM(get_paidsum(oper_id))`; в XLSX "Сумма оплаты" выводится напротив счета, если `veda_schets.f_operid = oper_id`.

## 9. Маппинг актов, УПД и расходов

### 9.1 Исходящие акты ERP -> Реализация товаров и услуг 1С

ERP:

- `veda_akts`;
- `veda_akts_details`;
- `veda_akts_details_opers`.

Классификация входящих документов по `veda_akts.f_type`:

| `f_type` | ERP-наименование | Канонический `document_type` | Блок REST 1С |
|---:|---|---|---|
| `7` | Поступление (акт, накладная) | `purchase` | `purchases` |
| `23` | АКТ от иностранного поставщика | `purchase` | `purchases` |
| `24` | Затраты иностранного партнера | `purchase` | `purchases` |
| `25` | Товар от иностранного поставщика | `purchase` | `purchases` |

Эти типы запрещено классифицировать как `sale`. Например, ERP-акт `0ЛБП-003165` от `06.06.2026` на `2 026,31 USD` (`veda_akts.f_type=23`) ищется в `purchases`, где 1С возвращает его на договоре поставщика `БП-053039`. Договор поставщика является контекстом документа и не должен подменяться кодом договора покупателя поставки.

Принадлежность такого документа поставке определяется не договором поставщика, а цепочкой `veda_akts -> veda_akts_details -> veda_akts_details_opers.f_operid -> veda_spec_invoices.f_specid`. Если операция поставщика является дочерней, сервис также показывает `veda_spec_invoices.f_parentid` как ссылку на клиентскую операцию. Входящий акт подтверждает расход поставщика и полноту обмена ERP ↔ 1С, но не заменяет исходящий закрывающий документ клиенту и не прибавляется повторно к клиентскому сальдо.

1С:

- `Документ.РеализацияТоваровУслуг`;
- чтение через `getAkts(dtb, dte, orgid, type)`, где `type != 1`;
- запись описана функцией `addupdAktR`.

Поля 1С из `getAkts`:

- `orgid`;
- `orgname`;
- `contrid`;
- `contrname`;
- `isdel`;
- `vidoper`;
- `isprov`;
- `num`;
- `dt`;
- `sum`;
- `val`;
- `dogid`;
- `com`;
- `dop`.

Поля, которые записывает `addupdAktR`:

- `Номер`;
- `Дата`;
- `Организация`;
- `Контрагент`;
- `ДоговорКонтрагента`;
- `СуммаДокумента`;
- `ВалютаДокумента`;
- `ПодразделениеОрганизации`;
- табличные части `АгентскиеУслуги`, `Услуги`, `Товары`.

Классификация строк `addupdAktR`:

| ERP `usl.izvozm` | Табличная часть 1С | Бухгалтерский смысл |
|---|---|---|
| `1` | `АгентскиеУслуги` | возмещаемые/агентские расчеты через 76.06 |
| `2` | `Услуги` | невозмещаемая услуга агента / выручка агента |
| `3` | `Товары` | товарная реализация |

Сравнивать:

| ERP | 1С | Правило |
|---|---|---|
| `veda_akts.f_kod1c` | `num` | код/номер документа 1С; сам по себе не является уникальным ключом сверки |
| `veda_akts.f_dt1c` или дата акта | `dt` | дата 1С-документа; обязательная часть ключа сверки |
| код 1С организации | `orgid` | обязательное совпадение |
| код 1С ЮЛ | `contrid` | обязательное совпадение |
| код 1С договора | `dogid` | обязательное совпадение для полного match |
| сумма документа/строки | `sum` или строка табличной части | контроль суммы |
| валюта | `val` | контроль валюты |

Ключ полного сопоставления акта/реализации: `тип документа + veda_akts.f_kod1c + veda_akts.f_dt1c + dogid`.
Если `f_dt1c` пустая или равна нулевой дате, использовать дату ERP-акта как fallback, но такой результат считать менее надежным и логировать как fallback-date.
Матч только по `f_kod1c` запрещен из-за дублей.

Если один акт ERP распределен между несколькими поставками, сумму поставки брать из `veda_akts_details_opers.f_sum`.

Для агрегирующего счета покупателю (`veda_schets.f_maininv`) ставка НДС шапки не является приоритетной. Если все активные дочерние счета имеют одинаковую `f_nds`, для сверки используется их единая ставка. Если ставки дочерних счетов различаются, НДС проверяется построчно; общий заголовок не должен создавать ложное расхождение. Контрольный кейс: `ВЛ-001806` и `ВЛ-001942` имеют в агрегирующей шапке `Без НДС`, но дочерние счета и строки 1С имеют `0%` — результат должен быть `match`.

## 10. Классификация возмещаемости

В ERP для матрицы нужны два денежных блока:

- `Возмещаемые расходы`;
- `Невозмещаемые расходы`.

Для ERP-строк использовать бизнес-классификацию ERP:

| ERP-признак | Значение в матрице |
|---|---|
| `isvozm = 1` | возмещаемые расходы |
| `isvozm = 2` | невозмещаемые услуги агента / агентское вознаграждение |
| `isvozm = 0` или пусто | классифицировать по типу операции и текущим ERP-правилам |

Важно: `isvozm = 2` не означает, что входящая 1С-строка `Услуги` должна попасть в клиентский взаиморасчет. Входящие `Услуги` часто являются нашими расходами подрядчика. Клиентский контекст в таком случае есть в ERP-распределении, а не в бухгалтерской аналитике строки 1С.

## 11. Ограничение по 1С-вкладке `Услуги`

Для входящих документов 1С строка `Услуги` может содержать только затратную аналитику:

- номенклатура;
- содержание;
- счет затрат;
- подразделение;
- субконто затрат.

Такая строка не обязана содержать договор с клиентом или договор-заявку. Если договор-заявка в строке отсутствует, то:

- строку можно использовать для подтверждения суммы;
- нельзя считать ее полным совпадением по поставке;
- нельзя подставлять клиентский договор как бухгалтерское основание строки.

Пример ограничения:

| Объект | Значение |
|---|---|
| документ 1С | `00БП-013350` |
| номенклатура 1С | `Транспортно-экспедиционные услуги` |
| содержание 1С | `Оформление документов VEDU4000192, коносамент NNLNSA251603424` |
| счет затрат | `20.01` |
| подразделение | `ОП Нск` |
| субконто | `Агентское вознаграждение / Услуги сторонних организаций` |
| строки 1С | `1980.91`, `1333.16`, `1685.93` |
| ERP-распределение | заявки `1063`, `1064`, `1065` через `veda_akts_details_opers` |
| тип ERP-операции | `Портовые расходы. Поручение на оплату` |
| возмещаемость ERP | `Невозмещаемые услуги Агента (Вознаграждение)` |

Статус по таким строкам:

```text
Подтверждено строкой 1С, нет договора-заявки в 1С
```

Это не ошибка услуги. Это ограничение бухгалтерской аналитики 1С для вкладки `Услуги`.

## 12. Где возможен полный match

Полный `ОК` возможен, если одновременно совпали:

- тип документа;
- код/номер 1С;
- дата;
- организация;
- ЮЛ/контрагент;
- сумма;
- валюта;
- договор 1С;
- договор-заявка или спецификация 1С, если сверяем поставку;
- строка табличной части, если документ разделен между поставками.

Для 1С-вкладки `АгентскиеУслуги` полный match возможен, если строка содержит:

- комитента;
- договор с комитентом;
- сумму строки;
- связь с документом;
- договор-заявку или код договора поставки.

Пример:

| Объект | Значение |
|---|---|
| документ 1С | `00БП-013275` |
| заявки ERP | `1063`, `1064`, `1065` |
| суммы | `12116.83`, `8154.70`, `10312.47` |
| ERP-источник суммы | `veda_akts_details_opers.f_sum` |
| 1С-источник | строка `АгентскиеУслуги` с договором комитента |

## 15. Статусы сверки

Backend-статусы:

| Статус | Когда ставить |
|---|---|
| `MATCH` | полное совпадение по ключам, суммам и договору/поставке |
| `MISSING_ERP_INVOICE` | у операции с ненулевой реализацией не выставлен либо не привязан счет покупателю; сравнивать с 1С пока нечего |
| `ERP_INVOICE_LINK_MISSING` | прямой связи операции со счетом нет, но в той же поставке найден единственный счет-кандидат той же суммы на другой операции; счет проверяется отдельно |
| `MISSING_ERP_CLOSING_DOCUMENT` | у операции с ненулевой реализацией отсутствует связь с закрывающим документом ERP; код и дата 1С еще не сформированы |
| `NOT_FOUND_IN_1C` | ERP-документ имеет ключ, но в 1С не найден |
| `ERP_CODE1C_MISSING` | ERP-документ существует, но код 1С не заполнен; номер ERP не используется как замена кода 1С |
| `NOT_LINKED_TO_DELIVERY_IN_ERP` | 1С-документ возвращен в области сверки, но не связан с выбранной поставкой ERP |
| `NOT_FOUND_IN_ERP` | После отдельного глобального поиска документ 1С отсутствует во всех исходных таблицах ERP |
| `SUM_MISMATCH` | документ найден, сумма отличается |
| `CONTRACT_MISMATCH` | документ найден, но договор/поставка отличаются |
| `NOT_COMPARABLE` | нет надежного ключа для сверки |
| `CONFIRMED_BY_1C_LINE_WITHOUT_CONTRACT` | сумма строки 1С подтверждена, но в строке нет договора-заявки |

UI-бейджи:

| Backend | UI |
|---|---|
| `MATCH` | `ОК` |
| `MISSING_ERP_INVOICE` | `Нет связи со счетом` |
| `ERP_INVOICE_LINK_MISSING` | `Счет на другой операции` |
| `MISSING_ERP_CLOSING_DOCUMENT` | `Нет закрывающего документа` |
| `NOT_FOUND_IN_1C` | `Не найдено в 1С` |
| `ERP_CODE1C_MISSING` | `Нет кода 1С в ERP` |
| `NOT_LINKED_TO_DELIVERY_IN_ERP` | `Нет связи с поставкой ERP` |
| `NOT_FOUND_IN_ERP` | `Не найдено в ERP` |
| `SUM_MISMATCH` | `Сумма расходится` |
| `CONTRACT_MISMATCH` | `Договор расходится` |
| `NOT_COMPARABLE` | `Нет ключа 1С` |
| `CONFIRMED_BY_1C_LINE_WITHOUT_CONTRACT` | `Подтверждено строкой 1С` |

Правила отсутствия выгрузки:

1. Если у операции ERP отсутствует счет или закрывающий документ, это не статус `NOT_FOUND_IN_1C`. ERP еще не сформировала полноценный объект обмена, поэтому сравнение с 1С невозможно.
2. `ERP_CODE1C_MISSING` ставится, когда ERP-документ существует, но код 1С пуст. Этот статус не доказывает отсутствие документа в 1С: точечная проверка невозможна.
3. `NOT_FOUND_IN_1C` ставится только тогда, когда в ERP существует полноценный документ с типом, кодом 1С и датой 1С, но REST 1С не вернул соответствующий документ.
4. Непарный документ из ответа 1С сначала получает `NOT_LINKED_TO_DELIVERY_IN_ERP`. Затем выполняется глобальный поиск в исходных таблицах ERP по типу, коду 1С и дате.
5. `NOT_FOUND_IN_ERP` ставится только после отрицательного результата глобального поиска. Документ, найденный в ERP у другой поставки, сохраняет статус `NOT_LINKED_TO_DELIVERY_IN_ERP`.
6. Нулевой документ 1С без пары не считается доказанным отсутствием в ERP и получает `NOT_COMPARABLE`.

Текст для `CONFIRMED_BY_1C_LINE_WITHOUT_CONTRACT` в детализации:

```text
Сумма совпала со строкой 1С, но строка не содержит договора-заявки. Полный ОК невозможен.
```

## 16. Требования к read-only данным 1С

Текущие функции 1С можно использовать там, где их полей достаточно:

- `getClients` - контрагенты, договоры, банковские счета;
- `getOrgs` - организации;
- `getAcchist` - банковские документы и расшифровка платежа;
- `getAkts` - документы реализации/поступления на уровне шапки;
- `getAktsEDO` / `getEDOArh` / `getEDPacket` - ЭДО;
- `getcoacsu` / `getcoacsuinfo` - акты сверки.

Для полной сверки поставок нужна доработка чтения актов/УПД:

1. Возвращать строки табличных частей `Услуги`, `АгентскиеУслуги`, `Товары`.
2. Возвращать код договора строки, если он есть.
3. Возвращать договор комитента для `АгентскиеУслуги`.
4. Возвращать затратные субконто для `Услуги`.
5. Возвращать сумму, НДС, содержание и номенклатуру строки.
6. Возвращать признак, по которому можно понять, есть ли в 1С договор-заявка.

Минимальный формат строки 1С для ERP:

```json
{
  "doc_kind": "purchase_act",
  "doc_num": "00БП-013350",
  "doc_date": "2025-08-05",
  "orgid": "000000001",
  "contrid": "...",
  "dogid": "...",
  "line_no": 1,
  "line_kind": "Услуги",
  "nomenclature": "Транспортно-экспедиционные услуги",
  "content": "...",
  "sum": 1980.91,
  "vat_rate": "0%",
  "expense_account": "20.01",
  "department": "ОП Нск",
  "subconto1": "Агентское вознаграждение",
  "subconto2": "Услуги сторонних организаций",
  "committent_contract_code": "",
  "spec_contract_code": ""
}
```

Если `spec_contract_code` пустой, ERP может подтвердить сумму, но не должна ставить полный `MATCH`.

## 17. Алгоритм сверки

1. Получить ERP-контекст по `spec_id`, договору, ЮЛ или клиенту.
2. Построить дерево `veda_contacts -> veda_clients -> veda_dogs -> veda_specs`.
3. Получить ERP-документы:
   - счета;
   - оплаты;
   - акты/УПД;
   - распределения актов по операциям;
   - распределения оплат по документам.
4. Получить 1С-данные по тем же организациям и периоду:
   - контрагентов и договоры;
   - счета;
   - банковские документы;
   - акты/УПД;
   - строки актов/УПД;
   - акт сверки, если нужен.
5. Нормализовать 1С-документы в общий формат:
   - `kind`;
   - `code1c`;
   - `date1c`;
   - `org_code`;
   - `contr_code`;
   - `dog_code`;
   - `spec_contract_code`;
   - `tax_invoice_number` и `tax_invoice_date` для реализации;
   - `sum`;
   - `line_kind`;
   - `line_sum`.
6. Выполнить match по ключам.
7. Проверить суммы.
8. Проверить договор/поставку.
9. Проставить статусы.
10. Сформировать матрицу и раскрытие документов.

## 18. Acceptance criteria

1. Экран строит иерархию через `veda_contacts -> veda_clients -> veda_dogs -> veda_specs`.
2. Для клиента `veda_contacts.f_id = 1190` должны отображаться все ЮЛ из `veda_clients.f_contactid = 1190`.
3. Счета сверяются с `Документ.СчетНаОплатуПокупателю` по номеру/дате/организации/контрагенту/договору/сумме.
4. Оплаты сверяются с `getAcchist`; сумма поставки в матрице берется из `SUM(get_paidsum(oper_id))`, а не из полной суммы выписки. `veda_acchist_docs.f_clssum` используется для аудита распределений и связи платежа с операцией.
5. Акты и УПД сверяются с `getAkts`; при распределении использовать `veda_akts_details_opers.f_sum`.
6. Для строк 1С `АгентскиеУслуги` с договором комитента возможен полный `ОК`.
7. Для строк 1С `Услуги` без договора-заявки ставится `Подтверждено строкой 1С`, а не полный `ОК`.
8. Документ `00БП-013350` должен подтверждать суммы `1980.91`, `1333.16`, `1685.93`, но не получать полный `ОК` без договора-заявки в строке 1С.
9. Документ `00БП-013275` должен иметь возможность полного match по строкам `АгентскиеУслуги`, если 1С возвращает договор комитента/договор поставки.
10. ERP-строка без надежного кода/даты 1С получает `Нет ключа 1С`.
