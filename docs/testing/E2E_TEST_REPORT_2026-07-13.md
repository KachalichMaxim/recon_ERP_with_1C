# E2E проверка ERP ↔ 1C

> Актуальный приемочный результат находится в разделе «Финальный контрольный запуск» в конце документа. Предыдущие секции сохранены как журнал обнаружения и устранения дефектов.

Дата проверки: 2026-07-13  
Контур: локальный Python backend через VPN `ppp0`  
ERP: MariaDB `veda25`  
1C: read-only REST API `/vedagent_dev/hs/reconciliation/v1`

## Условия запуска

Для локальной проверки потребовалась явная привязка исходящего адреса к VPN:

- `RECON_ERP_DB_BIND_ADDRESS=10.54.10.53`
- `RECON_ONEC_REST_BIND_ADDRESS=10.54.10.53`

Без bind-адреса macOS маршрутизировала `10.54.1.*` через другой tunnel, из-за чего MariaDB теряла соединение, а 1C возвращала пустой HTTP-ответ.

## Проверенный пользовательский маршрут

1. Открыть сервис.
2. Проверить статус подключений.
3. Найти клиента по тексту.
4. Найти договор по номеру.
5. Получить матрицу взаиморасчетов.
6. Выгрузить матрицу в XLSX.
7. Запустить сверку ERP ↔ 1C по поставке.
8. Выгрузить результат сверки в XLSX.
9. Сохранить комментарий к расхождению.

Что было исправлено в backend:

1. `veda_akts.f_type=7` больше не трактуется как `sale`; такие строки идут в `purchase`.
2. Физические документы `veda_akts` больше не схлопываются в один документ на операцию.
3. Документы `veda_akts` с `f_operid=0` добираются через связь `veda_akts_details.f_aktid -> veda_akts_details_opers.f_operid`.
4. Пустые нулевые строки `sale` без физического документа больше не добавляются в сверку.
5. Поле `number` не считается расхождением, если совпал `code1c`, потому что для платежей и поступлений ERP хранит входящий номер/номер ПП, а 1C REST сейчас возвращает внутренний номер документа. Для полноценной проверки номера нужен отдельный DTO-поле `incoming_number`.

Оставшееся ограничение backend:

- один физический документ поставщика может быть распределен в ERP на несколько поставок через `veda_akts_details_opers`;
- ERP для одной поставки видит долю документа, а шапка документа 1C содержит полную сумму;
- REST 1C теперь возвращает реальные строки, но Python matching еще должен сопоставлять `veda_akts_details_opers.f_sum` с `document_lines[].amount` и обрабатывать неоднозначные одинаковые суммы;
- REST 1C по `buyer_contract_code/committent_contract_code` не возвращает поставщицкие документы на сторонних договорах, хотя прямой поиск по `document_code` их находит; backend временно добирает их отдельными запросами по известным кодам.

Проверка обязательных полей `document_id`, `document_type`, `document_number`, `line_id`, `amount` прошла для всех перечисленных строк. Также возвращаются `content`, `nomenclature`, `line_kind`, `cost_account`, `vat_rate`, `vat_amount` и `contract_code`.

Запрос без `include` теперь отвечает `200 OK` и содержит `document_lines`, но одновременно возвращает `4492` договора и `10177` остатков даже при фильтре одного документа. Для интерактивной сверки нужно всегда задавать явный `include`, иначе ответ избыточен.

Остается несовпадение реализации с OpenAPI: повторяемые параметры `include=purchases&include=document_lines` обрабатываются как один первый блок. Оба блока возвращаются только при `include=purchases,document_lines`. Нужно либо поддержать `style=form, explode=true`, либо изменить OpenAPI на строку со списком через запятую.

## Проверка 1C REST

Работает:

- `GET /health`
- `GET /snapshot` по одному `document_code`
- `GET /snapshot` по `buyer_contract_code`
- `GET /snapshot` по `committent_contract_code`
- `GET /snapshot` по обоим договорным кодам

Ограничения текущей реализации 1C REST:

1. Повторяемый `contract_code` возвращает `500`, хотя MVP OpenAPI разрешает массив.
2. Несколько `include` в одном запросе фактически возвращают только первый блок. Backend временно обходит это несколькими короткими GET-запросами.
3. Длинный query string возвращает IIS `404.15`.
4. Ошибки возвращаются как HTML/text, а не JSON.
5. У успешного ответа пустой `Content-Type`.
6. Валюта приходит как `руб.`, желательно `RUB`.
7. `source_system` приходит как `1с`, желательно `1c`.
8. Номера документов приходят с хвостовыми пробелами, backend их trim-ит.

## Требуемые доработки P0

1. Применить SQL-схему `sql/1c_reconciliation_schema.sql` в MariaDB:
   - `veda_reconciliation_runs`
   - `veda_reconciliation_items`
   - `veda_reconciliation_comments`

2. Убрать синхронный `include_total=1` из основного пользовательского маршрута матрицы:
   - сейчас широкий период падает по timeout;
   - total/summary нужно считать асинхронно, кэшировать или показывать отдельно;
   - UI должен грузить первую страницу без total.

3. Для production обеспечить сетевой маршрут с backend-сервера до:
   - ERP MariaDB;
   - 1C REST API.
   
   Локальные `*_BIND_ADDRESS` не являются production-решением.

4. Разделить статусы `not_found_in_1c`:
   - нет `code1c` в ERP;
   - документ не выгружен в 1C;
   - документ есть в 1C, но на другом договоре;
   - документ есть, но сумма отличается.

5. Исправить 1C REST ответы по MVP-контракту:
   - JSON errors;
   - `Content-Type`;
   - валюта `RUB`;
   - поддержка повторяемого `contract_code` или явный отказ от него в OpenAPI.
