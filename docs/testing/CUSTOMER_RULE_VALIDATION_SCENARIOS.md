# Сценарии согласования правил сверки ERP и 1С

Дата подготовки: 16.07.2026  
Назначение: совместный проход с заказчиком, бухгалтерией и разработчиками ERP/1С до фиксации окончательных бизнес-правил.

## 1. Что уже реализовано без хардкода

В production-коде нет специальных условий для `spec_id=23687`, операций `451088`, `457606`, документов `0ЛБП-000788` или договора `БП-078924`. Эти идентификаторы используются только в автоматических тестах и документации как воспроизводимые примеры.

Общий алгоритм работает для любой поставки, если источники дают необходимые связи:

1. ERP-операции поставки читаются из исходных таблиц.
2. Счета, реализации, поступления и платежи связываются с операциями через таблицы распределений.
3. Один агрегированный документ ERP сохраняется в разрезе операций до сопоставления.
4. Документ 1С сопоставляется по виду документа, коду 1С и дате.
5. Если сумма шапки не равна доле поставки, проверяются `document_lines` или `payments[].allocations`.
6. Строка 1С выбирается автоматически только при однозначном совпадении суммы и, при наличии, договора строки.
7. Другой договор документа не является ошибкой, если связь строки с поставкой доказана.
8. Если подходящих строк несколько, сервис не выбирает одну случайно и возвращает неоднозначность.

Текущий фикс покрывает класс **«агрегированный акт/поступление содержит строки нескольких операций или поставок»**, а не только один известный документ.

## 2. Границы автоматического решения

| Уровень | Условие | Решение сервиса |
|---|---|---|
| Доказанное совпадение | Однозначная пара по виду, коду 1С, дате и сумме шапки либо строки/распределения | `match` |
| Доказанное расхождение | Пара однозначно найдена, но различаются сумма, валюта, дата, номер или НДС | Соответствующий статус расхождения |
| Предположение | В поставке найден единственный счет той же суммы и валюты, но он привязан к другой операции | `erp_invoice_link_missing`, счет показывается кандидатом, но не считается совпавшим |
| Невозможно проверить | Нет кода 1С, закрывающего документа, счета или однозначной строки распределения | Отдельный статус причины; `match` запрещен |

## 3. Приемочный маршрут

Для каждого сценария заказчик выполняет одинаковый маршрут:

1. Открывает поставку по ссылке ERP и проверяет операции, счета и закрывающие документы.
2. Запускает на странице поставки `Сверить с 1С`.
3. Открывает проблемную строку и переходит по ссылкам на ERP-документ и операцию.
4. При необходимости открывает соответствующий документ 1С.
5. Сверяет ожидаемый статус из таблиц ниже с фактическим.
6. Выбирает причину разбора, оставляет комментарий и сохраняет его.
7. В таблице решений фиксируется подтвержденное бизнес-правило.

## 4. Основные сценарии документов

| ID | Сценарий и контрольный пример | Текущее ожидаемое поведение | Решение заказчика |
|---|---|---|---|
| DOC-01 | Один документ ERP точно соответствует одному документу 1С. Контроль: `spec_id=17079` или `21802` | `match`; совпадают вид, код 1С, дата, сумма, валюта и НДС | Подтвердить состав обязательных реквизитов |
| DOC-02 | Агрегированная реализация содержит несколько клиентских операций. Контроль: `00БП-003225`, строки `365,00`, `3 283,00`, `7 661,00`, `3 327,84`, итого `14 636,84` | Каждая ERP-операция сопоставляется со своей строкой `document_lines`; шапка не дублируется на каждую операцию | Подтвердить, что строка реализации является достаточным доказательством |
| DOC-03 | Входящий агрегированный акт поставщика распределен между несколькими операциями/поставками. Контроль: `0ЛБП-000788`, шапка `26 000,00`; для `spec_id=23687` строки `14 798,04` и `3 523,34` | Две строки поставки дают `18 321,38`; прочие строки документа не включаются в ее сальдо; обе пары `match` | Подтвердить расчет только по распределенным строкам |
| DOC-04 | У 1С нет строк, но несколько ERP-строк в сумме точно равны одной шапке 1С. Контроль: шесть строк `00БП-002193`, итого `383 272,00` | ERP-строки объединяются только при единственной шапке и точном равенстве общей суммы | Подтвердить допустимость суммарного совпадения без детализации 1С |
| DOC-05 | Частичная оплата одной поставки внутри общего банковского документа. Контроль: `00БП-010591`, доля `3 765,79` | Сравнивается `payments[].allocations`, а не полная сумма шапки платежа | Подтвердить распределение платежа как источник суммы поставки |
| DOC-06 | В ERP есть реализация и счет, но счет находится на другой операции той же поставки. Контроль: операция `451088`, реализация `0ЛБП-000427`, кандидат счета `ВЛ-000576` на операции `457606`, сумма `5 937,54` | Реализация сверяется отдельно. Для счета выводится `Счет на другой операции`; это предупреждение, не `match` и не `Нет счета ERP` | Решить: исправлять связь в ERP или считать уникальный кандидат достаточным |
| DOC-07 | По расчетной операции нет ни прямого счета, ни единственного кандидата | `missing_erp_invoice` | Подтвердить, что отсутствие счета является блокирующей ошибкой полноты ERP |
| DOC-08 | `get_realizsum` возвращает сумму, но операция не связана с исходящим закрывающим документом | `missing_erp_closing_document`; сальдо помечается `not_comparable` | Подтвердить, что расчетная сумма без реализации не доказывает дебетовый оборот 1С |
| DOC-09 | ERP-документ есть, но код 1С пуст. Контроль: акт `комиссия попова`, `92 250,83 RUB` | `erp_code1c_missing`; номер ERP не подставляется вместо кода 1С | Подтвердить обязательность кода 1С для точечной проверки |
| DOC-10 | Код 1С заполнен, но документ не найден в ответе 1С. Контроль: `0ЛБП-001309`, `14 137,00 RUB`, `spec_id=23832` | `not_found_in_1c` только после проверки вида, кода, даты и безопасного поиска по сумме/валюте | Подтвердить владельца исправления обмена |
| DOC-11 | Документ 1С существует в ERP глобально, но не связан с выбранной поставкой. Контроль: платеж `0ЛБП-001741`, `75 982,50`, `spec_id=23920` | `not_linked_to_delivery_in_erp`, а не `not_found_in_erp` | Подтвердить, что связь с поставкой обязательна |
| DOC-12 | Документ пришел из поставочной области 1С, но глобально не найден в ERP | `not_found_in_erp` | Проверить, какие типы 1С действительно должны иметь ERP-источник |
| DOC-13 | Документ 1С проведен на договоре поставщика/связанном договоре, но строка однозначно относится к операции поставки | `match`; договор показывается справочно и не формирует `contract_mismatch` | Подтвердить, что договор является контекстом выбора, а не самостоятельной ошибкой |
| DOC-14 | Документ удален или не проведен в 1С | Предупреждение/расхождение состояния; документ не считается полноценным подтверждением учета | Определить правила для записанного, проведенного и помеченного на удаление документа |
| DOC-15 | Для одной ERP-строки найдены две одинаковые строки 1С | `ambiguous_match`; автоматический выбор запрещен | Определить дополнительный стабильный ключ либо оставить ручной разбор |

## 5. Сценарии реквизитов

| ID | Сценарий и контрольный пример | Текущее ожидаемое поведение | Решение заказчика |
|---|---|---|---|
| ATR-01 | Сумма различается. `ВА-011606`: ERP `280 687,18 RUB`, 1С `275 687,18 RUB` | `amount_mismatch` | Подтвердить нулевой допуск либо иной порог |
| ATR-02 | Валюта различается либо одна сторона хранит пересчитанную рублевую сумму | `amount_mismatch` с причиной `валюта`; суммы разных валют не складываются | Определить сравнение исходной валюты, валюты взаиморасчетов и рублевого эквивалента |
| ATR-03 | Дата различается. `0ЛБП-000770`: ERP `24.07.2024`, 1С `19.09.2024`, сумма `88 020,60` | Пара по коду и сумме находится, результат `date_mismatch` | Подтвердить, какая дата является учетной и допустим ли сдвиг |
| ATR-04 | Номер отличается, но код 1С, дата и сумма совпадают | `number_mismatch` | Решить, является ли номер блокирующим либо информационным реквизитом |
| ATR-05 | НДС `0%` на одной стороне и `Без НДС` на другой | Сейчас различие фиксируется, если нормализация не доказывает эквивалентность | Решить, считать ли `0%` и `Без НДС` разными во всех типах документов |
| ATR-06 | Реальная разница НДС. `00БП-009290`: ERP `0%`, 1С `20%`; `ВЭ-000278`: ERP `Без НДС`, 1С `5%` | `vat_mismatch` | Подтвердить ставку как обязательный реквизит |
| ATR-07 | Разница `0,01` из-за округления. Контроль: `spec_id=24282` | Сейчас `amount_mismatch`/разница сальдо `0,01` сохраняется | Утвердить допуск: `0`, `0,01` либо иной порог по валюте |
| ATR-08 | Агрегирующий счет имеет пустую/общую ставку, дочерние счета имеют единую ставку. Контроль: `spec_id=29520` | Используется единая ставка дочерних счетов; ложный `vat_mismatch` не создается | Подтвердить приоритет ставки дочерних документов |

## 6. Сценарии сальдо

Формула ERP: `SUM(get_paidsum) - SUM(get_realizsum)` по операциям поставки. Возмещаемые и невозмещаемые расходы являются расшифровкой реализации, а не отдельным прибавлением к сальдо.

| ID | Сценарий | Текущее ожидаемое поведение | Решение заказчика |
|---|---|---|---|
| BAL-01 | Документы совпали и сальдо совпало | Результат `Подтверждено` | Подтвердить зеленый итог |
| BAL-02 | Сальдо равно, но есть ошибки полноты/реквизитов | Итог не должен становиться полностью зеленым только из-за нулевой дельты | Утвердить текст: `Сальдо совпало, есть расхождения документов` |
| BAL-03 | Сальдо различается и разница объяснена конкретным платежом/реализацией | Показываются дельта и причинные строки | Подтвердить приоритет причин по сумме влияния |
| BAL-04 | Сальдо ERP рассчитано, но закрывающие документы отсутствуют | `not_comparable`, даже если числовая дельта случайно равна нулю | Подтвердить запрет статуса `Подтверждено` |
| BAL-05 | Документ связан с поставкой строкой, но проведен по другому договору | Строка участвует в сальдо поставки; прямой остаток договора и связанная корректировка показываются раздельно | Подтвердить способ представления бухгалтерского остатка |

## 7. Корректировки поставки — открытое правило

Текущий SQL-адаптер не читает `veda_corrects`. Поэтому нельзя утверждать, что поставки с корректировками уже покрыты.

Фактический контрольный набор:

| spec_id | Поставка | Базовый отчет | Корректировки | Состояние |
|---:|---|---:|---|---|
| `29749` | договор `ВЛ-0002`, заявка `13381` | `f_num=0`, статус `3` | `f_num=1` закрыта; `f_num=2` открыта | Требуется правило включения версий |
| `28182` | договор `ВА-1723`, спецификация `1` | `f_num=0` закрыта | `f_num=1` открыта | Требуется правило открытой корректировки |
| `28672` | договор `804`, спецификация `37` | `f_num=0` закрыта | `f_num=1` открыта | Требуется правило открытой корректировки |

Заказчик должен выбрать одно правило:

- `COR-A`: итог равен базовому отчету плюс все закрытые корректировки;
- `COR-B`: используется только последняя закрытая версия, предыдущие заменяются;
- `COR-C`: базовый отчет и каждая корректировка показываются отдельными слоями, итог подтверждается вручную.

До выбора правила сервис должен показывать предупреждение `Есть корректировки: автоматический итог не подтвержден`, а не зеленый результат.

## 8. Журнал и воспроизводимость

| ID | Проверка | Критерий приемки |
|---|---|---|
| LOG-01 | Сохранение запуска | Для каждого запуска сохраняются `run_id`, пользователь, поставка, время, версия правил, входные коды договоров, итоги и метрики источников |
| LOG-02 | Сохранение строки | Для каждого расхождения сохраняются статус, причина, ERP/1С идентификаторы и сравниваемые значения |
| LOG-03 | Разбор пользователем | Причина и комментарий сохраняются на backend и видны после повторного входа, а не только в браузере |
| LOG-04 | Повторный запуск | Новый запуск не перезаписывает старый; можно сравнить результаты до и после исправления |
| LOG-05 | Экспорт | XLSX воспроизводит выбранный запуск и содержит его номер/дату |

## 9. Нефункциональная приемка

| ID | Проверка | Критерий |
|---|---|---|
| NFT-01 | Одна поставка | p50 полного цикла не более `12 с`, p95 не более `18 с` при доступных ERP и 1С |
| NFT-02 | Массовая сверка | Договор/список поставок выполняется background runner, UI не держит один длинный HTTP-запрос |
| NFT-03 | Нагрузка на ERP | Операции читаются пакетно; тяжелые `get_paidsum/get_realizsum` не вызываются повторно для одной поставки |
| NFT-04 | Нагрузка на 1С | Пакетный запрос предпочтителен; точечные запросы ограничены по параллелизму и кэшируются |
| NFT-05 | Ошибка источника | Пользователь видит, какой источник недоступен; частичный ответ не превращается в бизнес-расхождение |
