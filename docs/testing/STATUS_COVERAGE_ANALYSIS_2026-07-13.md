# Аналитика полноты статусов сверки ERP и 1С

Дата проверки: 13.07.2026.

## 1. Цель и единица анализа

Проверка отвечает на два разных вопроса:

1. Готова ли операция ERP к выгрузке в 1С: есть ли у нее счет и закрывающий документ.
2. Совпадает ли уже сформированный ERP-документ с документом 1С.

Единица первой проверки - операция поставки ERP. Единица документной сверки - документ либо распределенная строка документа в рамках поставки.

Отсутствие связи операции с документом не равнозначно отсутствию документа в 1С. В этом случае ERP еще не сформировала надежный ключ сравнения.

## 2. Проверенные связи ERP

### Счет покупателю

Операция считается связанной со счетом, если выполнено хотя бы одно условие:

```sql
veda_schets.f_operid = operation_id
```

или существует распределение:

```sql
veda_schets_details_opers.f_operid = operation_id
-> veda_schets_details.f_id
-> veda_schets.f_id
```

Учитываются счета `veda_schets.f_type = 1`, кроме удаленных/отмененных записей `f_status = 9`.

### Закрывающий документ

Операция считается связанной с закрывающим документом, если выполнено хотя бы одно условие:

```sql
veda_akts.f_operid = operation_id
```

или существует распределение:

```sql
veda_akts_details_opers.f_operid = operation_id
-> veda_akts_details.f_id
-> veda_akts.f_id
```

Для агрегированных актов код и дата 1С берутся из родительского объекта `veda_akts.f_mainakt`, а сумма и операция сохраняются на уровне дочерней строки.

## 3. Результат проверки ERP

Проверена зрелая выборка поставок июля 2025 года: операции со статусом `7` и классификацией расходов `f_isvozm IN (1, 2)`.

| Показатель | Количество | Доля от 4 300 операций с ненулевой реализацией |
|---|---:|---:|
| Всего операций в выборке | 9 100 | - |
| Ненулевой `get_realizsum` | 4 300 | 100,00% |
| Нет связи с закрывающим документом | 887 | 20,63% |
| Нет связи со счетом покупателю | 2 131 | 49,56% |
| Нет обеих связей | 885 | 20,58% |

Эти показатели характеризуют трассируемость операции, а не сумму задолженности. Отсутствие счета на уровне операции может означать как реально невыставленный счет, так и счет, который не распределен до конкретной операции. Поэтому пользовательский текст: `Счет не выставлен или не привязан`.

Операции без связи со счетом: `361095`, `366479`, `374905`, `376233`, `376239`, `376754`, `377349`. Они не отменяют 19 подтвержденных совпадений и не меняют формулу сальдо, но показывают неполную трассируемость выставления счетов.

## 5. Проверка направления 1С -> ERP

Проверены пять поставок: `20334`, `20341`, `20348`, `20350`, `20351`.

Первичная сверка в пределах выбранных поставок нашла 13 документов 1С без пары. После глобального поиска в исходных таблицах ERP:

| Результат | Количество |
|---|---:|
| Документ существует в ERP, но связан с другой поставкой | 12 |
| Нулевой документ 1С, не участвующий в документной сверке | 1 |
| Подтверждено: документ есть в 1С и отсутствует в ERP глобально | 0 |

Следовательно, проверенная выборка не подтверждает реальный кейс `есть в 1С, нет в ERP`. Такой кейс технически возможен: ручной документ 1С, удаленная запись ERP или незавершенный обратный обмен. Он должен поддерживаться статусом `NOT_FOUND_IN_ERP`, но только после глобального поиска.

Глобальный поиск выполняется только для непарных документов 1С. На проверенных запусках он занимал `120-341 мс`; это время сохраняется в метрике `erp_global_lookup_ms`.

## 6. Полная статусная модель

| Статус | Бизнес-смысл | Условие |
|---|---|---|
| `MATCH` | Документ подтвержден | Совпали тип, код/номер, дата и сумма; договор отображается как контекст |
| `MISSING_ERP_INVOICE` | Счет не выставлен или не привязан | У операции есть ненулевая реализация, но нет прямой или распределенной связи со счетом |
| `MISSING_ERP_CLOSING_DOCUMENT` | Нет закрывающего документа | У операции есть ненулевая реализация, но нет прямой или распределенной связи с актом/реализацией |
| `NOT_FOUND_IN_1C` | Полноценный ERP-документ не найден в ответе 1С | ERP имеет тип, код 1С и дату 1С, но пары в REST 1С нет |
| `NOT_LINKED_TO_DELIVERY_IN_ERP` | Документ 1С есть в ERP у другой поставки | Пары в выбранной поставке нет, глобальный поиск ERP успешен |
| `NOT_FOUND_IN_ERP` | Документ существует только в 1С | Пары в поставке нет и глобальный поиск ERP отрицательный |
| `AMOUNT_MISMATCH` | Расходится сумма | Пара найдена, сумма отличается сверх допуска |
| `NUMBER_MISMATCH` | Расходится номер | Пара по надежным ключам найдена, номер отличается |
| `DATE_MISMATCH` | Расходится дата | Пара найдена по другим надежным ключам, дата отличается |
| `VAT_MISMATCH` | Расходится ставка/сумма НДС | Пара найдена, НДС ERP и 1С различается |
| `DUPLICATE_IN_1C` / `AMBIGUOUS_MATCH` | Нельзя однозначно выбрать документ | По ключу найдено несколько кандидатов |
| `CONTRACT_CONTEXT_MISSING` | Недостаточно аналитики поставки | 1С не вернула договор/строку, необходимую для распределения |
| `NOT_COMPARABLE` | Нет надежного основания сравнения | Нулевой документ или недостаточный набор ключей |

Договор документа не является самостоятельной ошибкой, если документ однозначно найден по коду, дате и сумме. В реальном учете связанный документ может быть проведен на другом договоре; этот договор показывается пользователю справочно.

## 7. Ограничение полноты обратной сверки

REST 1С сейчас возвращает документы по переданным кодам договоров и известным ERP-документам. Поэтому сервис надежно проверяет документы, которые можно обнаружить через эти ключи.

Для гарантированного выявления всех документов `только в 1С`, включая документы на связанных внешних договорах, 1С в `mode=delivery_reconciliation` должна учитывать коды договоров в шапке, табличных частях и расшифровках платежей и вернуть все связанные документы независимо от того, известен ли их код ERP. Отдельный параметр `scope=delivery` не требуется. До такой доработки отсутствие выявленного `NOT_FOUND_IN_ERP` означает `не найдено в проверенной области`, а не математическое доказательство отсутствия кейса во всей 1С.

## 8. Автоматические контроли

1. Для каждой операции с ненулевым `get_realizsum` проверять наличие счета и закрывающего документа.
2. Не преобразовывать отсутствие ERP-связи в `NOT_FOUND_IN_1C`.
3. Для каждого непарного документа 1С выполнять глобальный поиск ERP по `kind + code1c + date + amount`.
4. Нулевые документы не учитывать как глобально отсутствующие.
5. Сохранять статус, причину, идентификатор операции, ссылки на ERP-объекты и длительности чтения ERP/1С в журнале запуска.
