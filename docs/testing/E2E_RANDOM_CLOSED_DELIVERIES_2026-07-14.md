# E2E-проверка ERP ↔ 1С на случайной выборке закрытых поставок

Дата проверки: 14.07.2026  
Контуры: production MariaDB ERP и тестовый read-only REST 1С  
Метод: полный Python pipeline `ERP -> canonical model -> 1С REST -> matching -> контроль сальдо`.

## Цель и выборка

Проверка отвечает на два вопроса:

1. воспроизводит ли сервис документы и сальдо реально закрытых поставок;
2. какие расхождения остаются после контрольной заявки 1051.

Критерии отбора:

- дата поставки с `01.01.2024` по `31.12.2024`;
- заполнен код договора с покупателем `veda_specs.f_kod1cb`;
- заполнена дата закрытия `veda_specs.f_dtclose`;
- период документов ERP/1С с `01.01.2024` по `31.12.2025`;
- выбор `random.sample`, seed `20260714`.

Из `6 693` поставок периода критериям соответствовали `5 132`. Проверены 10 поставок:

`16836, 16708, 14286, 17079, 13558, 15375, 12311, 17049, 17451, 15782`.

Незакрытые поставки в выборку не входят. Дата закрытия берется только из `veda_specs.f_dtclose`; статус `veda_specs.f_status` сохраняется для диагностики, но не заменяет дату закрытия.

## Исправления по результатам проверки

До финального запуска устранены три подтвержденных дефекта Python-сервиса.

1. Платежи ERP ранее объединялись по операции. Теперь каждая строка `veda_acchist_docs` возвращается отдельным распределением платежа с суммой `f_clssum`; контрольное `SUM(f_clssum)` равно `get_paidsum`.
2. Одинаковый код и сумма при разных датах теперь дают `date_mismatch`. Пример: `0ЛБП-000770`, ERP `24.07.2024`, 1С `19.09.2024`, сумма `88 020,60 RUB`.
3. Если несколько дочерних ERP-строк образуют одну шапку 1С, сервис сравнивает их суммарно только при точном равенстве шапке. Если 1С отдает полноценные строки, операция остается детальной. Пример: шесть ERP-строк `00БП-002193` дают `383 272,00 RUB` и совпадают с одной агрегированной строкой 1С.

Дополнительно точечный поиск по коду теперь исключает документы других лет с тем же номером. Сначала выбирается точная дата; если дата различается, допускается только точное совпадение суммы и валюты. Это уменьшило ложные `not_linked_to_delivery_in_erp` с 18 до 2.

## Как трактовать оставшиеся статусы

### Ошибки полноты ERP

`missing_erp_closing_document` означает: операция ERP есть, сумма рассчитана `get_realizsum`, но операция не связана с закрывающим документом.  
`missing_erp_invoice` означает: по операции отсутствует связь со счетом покупателю.  

Это не ошибки REST 1С. Нечего сопоставлять по коду 1С, поэтому такие операции должны оставаться красными до исправления ERP-данных.

### Документ только в одной системе

`not_found_in_1c` — у ERP-документа есть идентификаторы, но REST 1С не вернул документ после проверки типа, кода, даты и безопасного fallback по сумме.  
`not_found_in_erp` — документ из поставочной области 1С не найден в исходных таблицах ERP по типу, коду 1С и дате.  
`not_linked_to_delivery_in_erp` — документ существует в ERP глобально, но ERP-связь с проверяемой поставкой отсутствует.

Оставшиеся два `not_linked_to_delivery_in_erp` — платежи `0КБП-004186` на `128,94 RUB` и `0КБП-000882` на `0,27 RUB`. Их нужно разбирать как связь платежа с поставкой, а не как отсутствие документа.

### Фактические различия реквизитов

Примеры подтвержденных расхождений НДС:

- счет `ВА-005623`: ERP `Без НДС`, 1С `20%`;
- поступление `00БП-009290`: ERP `0%`, строка 1С `20%`;
- счет `ВЭ-000278`: ERP `Без НДС`, 1С `5%`.

Пример расхождения суммы: платеж `0КБП-000844` имеет ERP-распределение `7 715,16`, а шапка 1С равна `812 030,00`; среди распределений 1С нет строки `7 715,16` на договор `БП-057430`. Это реальный кейс для бухгалтерского разбора.

`ambiguous_match` используется, когда в одном документе 1С несколько строк с одинаковой суммой и договором и API не содержит признака операции ERP. Сервис намеренно не выбирает строку случайно.

## Производительность

Измерение выполнено последовательно на 10 поставках, без кэша и без подмены источников.

| Этап | p50 | p95 |
|---|---:|---:|
| ERP MariaDB | 4,83 c | 4,95 c |
| REST 1С | 6,35 c | 11,72 c |
| Полный цикл одной поставки | 12,65 c | 17,81 c |
| 10 поставок последовательно | 130,72 c суммарно | 130,72 c суммарно |

Фактический p95 превышает 15 секунд. Одна поставка пригодна для интерактивного запуска с экраном прогресса, но массовую сверку нельзя выполнять синхронно в браузере. Для договора/периода нужен background runner с партиями, журналом прогресса и повторной обработкой только измененных документов.

Основное ограничение скорости 1С: массив `document_code` пока не возвращает все переданные документы, поэтому Python выполняет точечные REST-запросы параллельно с ограничением до четырех соединений. После исправления пакетного фильтра 1С нужно повторить замер.

## Итог по блокерам

Для сверки одной известной поставки функциональных блокеров REST 1С больше нет: шапки, строки, распределения платежей, связанные договоры, счета-фактуры и остатки читаются.

Остались:

1. бизнес-расхождения и неполнота ERP, которые сервис уже корректно показывает;
2. неоднозначные одинаковые строки без идентификатора операции — разрешаются ручным разбором либо дополнительной аналитикой 1С;
3. пакетный `document_code` и стандартный формат массива `include` — нужны для производительности массовой сверки;
4. массовый запуск должен выполняться background runner, а не одним live-запросом UI.
