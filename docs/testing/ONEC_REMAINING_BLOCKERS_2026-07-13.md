# Остаточные блокеры 1C REST для полной сверки ERP ↔ 1C

Дата проверки: 2026-07-13  
Контрольная поставка ERP: `spec_id=20334`, заявка `1051`, базовый договор `660/1`  
Договор покупателя 1C: `БП-068417`  
Договор комитента 1C: `БП-068418`

## Что уже работает

- `GET /health` и read-only Basic Auth.
- Поиск документа по одному `document_code`.
- Шапки счетов, оплат, реализаций и поступлений.
- `document_lines` содержит реальные строки счетов, реализаций и поступлений.
- Сумма строк совпадает с суммой шапки проверенных документов.
- `payments[].allocations` содержит распределение платежа по счетам.
- `balances` содержит входящие остатки, обороты и исходящие остатки по договорам.

## Контрольное сальдо заявки 1051

ERP относит к `spec_id=20334` не только документы на прямых договорах заявки, но и доли платежей/расходов с других договоров через операции и корректировки.

| Показатель ERP | Сумма, RUB |
| --- | ---: |
| Оплаты `SUM(get_paidsum)` | 203 091,35 |
| Возмещаемые расходы `SUM(get_realizsum)` | 171 364,16 |
| Невозмещаемые расходы `SUM(get_realizsum)` | 38 394,01 |
| Сальдо: оплаты − расходы | -6 666,82 |

1C по прямым договорам заявки на `31.08.2025` возвращает:

| Договор 1C | Исходящий дебет, RUB |
| --- | ---: |
| Покупатель `БП-068417` | 3 327,84 |
| Комитент `БП-068418` | 2 739,06 |
| Итого долг по прямым договорам | 6 066,90 |

Разница прямых остатков равна `599,92`, но это не отсутствующий документ. В ERP дочерний акт `ВА-009592/1` (`f_id=180226`) связан через `f_mainakt=180227` с агрегирующим актом `00БП-003228`. В 1C сумма `599,92` находится во второй строке реализации `00БП-003228` на договоре `БП-013397`.

Python-сервис разрешает агрегирующий акт и учитывает эту строку как связанную проводку на другом договоре:

| Расчет | Сумма, RUB |
|---|---:|
| Сальдо 1C по прямым договорам `БП-068417` и `БП-068418` | -6 066,90 |
| Связанная строка реализации на `БП-013397` | -599,92 |
| Сальдо 1C по поставке после распределения | -6 666,82 |
| Сальдо ERP | -6 666,82 |

Контроль сальдо закрыт на стороне Python. Для полного поиска документов 1C, которых нет в ERP, по-прежнему нужен серверный `scope=delivery` или эквивалентная связь внешнего договора/строки с заявкой.

## P1. Договор распределения платежа как справочный контекст

Платеж `00БП-010591` на `9 660,68` относится в шапке к заявке 1049 (`БП-068412`), но распределен на два счета:

- `5 894,89` → `ВА-007514`, заявка 1049, договор `БП-068412`;
- `3 765,79` → `ВА-007517`, заявка 1051, договор счета `БП-068418`.

REST сейчас возвращает `allocations[].contract_code=БП-068412` для обеих строк. Для второй строки это не соответствует договору целевого счета.

Аналогично платеж `00БП-010302` распределен на счета заявки 1051 с разными договорами:

- `ВА-007260` и `ВА-007261` — `БП-068418`;
- `ВА-007262` — `БП-068417`.

REST возвращает `БП-068418` для всех трех распределений.

Это не блокирует сверку наличия, реквизитов и суммы документа. Python считает однозначно найденное распределение совпавшим, а договор показывает справочно. Для точного отображения контекста желательно `payments[].allocations[].contract_code` брать из договора конкретной строки расшифровки платежа/целевого счета, а не из договора шапки платежа. `invoice_id` и `invoice_number` уже реализованы.

Критерий приемки: у распределения `3 765,79` платежа `00БП-010591` указан договор `БП-068418`; у распределения `11 309,00` платежа `00БП-010302` указан `БП-068417`.

## P0. Связанные поступления поставщиков не находятся по поставке

Запрос:

```http
GET /reconciliation/v1/snapshot?date_from=2025-07-01&date_to=2025-08-31&buyer_contract_code=БП-068417&committent_contract_code=БП-068418&include=purchases,document_lines
```

Фактически:

```json
{
  "purchases": [],
  "document_lines": []
}
```

При прямом поиске по `document_code` поступления находятся, например:

- `00БП-011330`;
- `00БП-012300`;
- `00БП-012856`;
- `00БП-012862`;
- `00БП-013452`.

Почему это блокер: Python может проверить известные ERP-документы в направлении ERP → 1C, но не может обнаружить поступление, которое существует в 1C и отсутствует в ERP.

Требование: при фильтре по договору покупателя/комитента вернуть все документы, которые 1C связывает с заявкой во вкладке «Документы», даже если прямой договор поступления является договором поставщика.

Для каждого такого документа вернуть:

- `contract_code` — прямой договор поставщика;
- `linked_contract_codes` — договор покупателя/комитента заявки;
- `spec_number` — номер заявки, если доступен.

Критерий приемки: запрос по `БП-068417`/`БП-068418` возвращает перечисленные связанные поступления без передачи их `document_code`.

## P0. Нет однозначной связи строки 1C с поставкой

Строки теперь корректны по структуре и суммам. Например:

- `00БП-012300`: строка `6 462,89` совпадает с `veda_akts_details_opers.f_sum` для `spec_id=20334`;
- `00БП-012856`: строка `7 661,34` совпадает с ERP для той же поставки.

Но строки не содержат `spec_number`, `linked_contract_code` или другой идентификатор заявки. Сейчас связь можно вывести только по сумме, содержанию, НДС и договору. При двух одинаковых строках результат будет неоднозначным.

Требование: вернуть в `document_lines[]` хотя бы одно поле связи с поставкой:

- `spec_number`; или
- `linked_contract_code`; или
- `allocations[]` с суммой и договором/заявкой.

Критерий приемки: строка `6 462,89` документа `00БП-012300` однозначно указывает на заявку `1051` либо на ее договор 1C.

Если такой аналитики в 1C физически нет, API должен вернуть предупреждение `DELIVERY_ALLOCATION_UNAVAILABLE`. Тогда Python фиксирует `AMBIGUOUS_MATCH`, а не зеленый статус.

## P0. Фильтр `spec_number` падает

Запрос:

```http
GET /reconciliation/v1/snapshot?date_from=2025-07-01&date_to=2025-08-31&spec_number=1051&include=purchases,document_lines
```

Фактически: `HTTP 500`, модуль 1C, строка 65, ошибка создания `HTTPСервисОтвет`.

Требование: либо реализовать фильтр и вернуть `200`, либо удалить параметр из OpenAPI и полностью обеспечить выборку поставки через `buyer_contract_code`/`committent_contract_code`.

## P0 для производительности. Массив `document_code` не работает

Повторяемый запрос:

```http
...&document_code=00БП-012300&document_code=00БП-012856&document_type=purchase
```

возвращает только первый документ. Значение через запятую не возвращает ни одного документа.

Требование: поддержать массив по OpenAPI (`style=form`, `explode=true`).

Критерий приемки: один GET по двум кодам возвращает две шапки и строки обоих документов. Без этого Python вынужден выполнять отдельный HTTP-запрос на каждый ERP-документ.

## P1. Формат `include` не соответствует OpenAPI

OpenAPI задает:

```text
include=purchases&include=document_lines
```

Фактически возвращается только первый блок. Работает нестандартный вариант:

```text
include=purchases,document_lines
```

Нужно выбрать один формат и синхронизировать реализацию с OpenAPI. До исправления Python может использовать вариант через запятую.

## P1. Не заполнены реквизиты счета-фактуры/УПД

Для реализации `00БП-003225` поля `invoice_number` и `invoice_date` пусты, хотя в 1C существует связанный «Счет-фактура выданный».

Если статус «Нет СФ» остается в объеме сверки, 1C должна:

- заполнить `invoice_number` и `invoice_date` в `sales[]`; или
- вернуть отдельный блок связанных счетов-фактур с `sale_id`/`sale_number`.

## P1. Избыточный ответ без `include`

Запрос одного документа без `include` отвечает `200`, но возвращает около `5,2 МБ`: `4492` договора и `10177` строк остатков. Время ответа — около `4,16 c`; с явным `include` — около `0,58 c` и `2,8 КБ`.

Это не блокирует Python: он будет всегда передавать явный `include`. Исправление желательно для защиты 1C от случайных тяжелых запросов.

## Что остается на стороне Python

На 13.07.2026 пункты 1-8 реализованы:

1. `document_lines`, `payments.allocations` и `balances` сохраняются в canonical model.
2. ERP-доли поступлений сопоставляются со строками 1C.
3. Одинаковые кандидаты дают `AMBIGUOUS_MATCH`.
4. Сальдо ERP сравнивается с суммой `closing_credit - closing_debit` по двум договорам заявки.
5. Частичные оплаты сопоставляются по `payments.allocations`.
6. Интерфейс/XLSX показывают основание matching, id строки и контроль сальдо.
7. Дочерние ERP-акты разрешаются через `veda_akts.f_mainakt`, а их суммы сопоставляются со строками агрегирующего документа 1C.
8. Сальдо поставки включает подтвержденные строки реализации и распределения платежей на связанных внешних договорах; прямое сальдо и корректировка показываются отдельно.

На стороне Python остается эксплуатационная работа:

- обеспечить сетевой маршрут production-сервера к REST 1C;
- после ответа разработчика 1C повторить E2E и зафиксировать приемку.

Выполнено 13.07.2026:

- таблицы `veda_reconciliation_runs`, `veda_reconciliation_items`, `veda_reconciliation_comments` созданы в production MariaDB;
- сохранение запуска и строк выполняется одной транзакцией;
- контрольный запуск `spec_id=20334` записан полностью: 1 шапка и 19 результатов;
- добавлен воспроизводимый CLI `tools/e2e_reconcile_delivery.py` с JSON/XLSX-выходом.

## Итог

Проверка известных ERP-документов уже выполняется end-to-end. Для контрольной заявки получено `19 match`, `0 mismatch`, `0 not_found`; итоговые сальдо ERP и 1C совпадают. Основные оставшиеся функциональные блокеры находятся в REST 1C:

1. корректный `contract_code`/`linked_contract_code` в каждой строке `payments.allocations`;
2. выборка связанных документов по поставке (`scope=delivery` или эквивалентный фильтр), чтобы не делать GET на каждый известный код;
3. явная связь одинаковых строк документа с заявкой, иначе результат остается `AMBIGUOUS_MATCH`;
4. `incoming_number`/`incoming_date`, если требуется проверять входящий номер отдельно от внутреннего номера 1C;
5. связанные реквизиты счета-фактуры, если статус «Нет СФ» входит в объем сверки.
