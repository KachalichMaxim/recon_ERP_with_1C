# Эталонные сценарии MVP

Сценарии используются для регрессионной проверки и пользовательских интервью. Статусы правил остаются экспериментальными до подтверждения владельцами процесса.

| ID | Ситуация | Ожидаемый результат | Объяснение пользователю | Следующее действие |
|---|---|---|---|---|
| CASE-01 | ERP-документ и документ 1С совпали | `match` | Документ найден, проверенные реквизиты совпали | Действие не требуется |
| CASE-02 | По клиентской операции нет прямой связи со счетом | `missing_erp_invoice`; при единственном найденном кандидате — `erp_invoice_link_missing` | Если в поставке есть единственный счет той же суммы на отдельной операции, он показывается кандидатом и проверяется собственной строкой; связь не считается доказанной | Проверить кандидата и при необходимости связать счет с исходной операцией ERP |
| CASE-03 | По операции нет закрывающего документа | `missing_erp_closing_document` | ERP не содержит ожидаемый закрывающий документ | Открыть операцию ERP и оформить/привязать документ |
| CASE-04 | В ERP есть документ с кодом 1С, но 1С его не вернула | `not_found_in_1c` | Документ не найден в доступной области 1С | Проверить обмен и достаточность scope/периода |
| CASE-05 | Пара найдена, даты различаются | `date_mismatch` | Документы найдены, но дата отличается | Сверить даты документов в ERP и 1С |
| CASE-06 | Оплата распределена частично | `match` или `amount_mismatch` по строкам распределения | Сравнение выполнено по распределениям, а не только по шапке платежа | Проверить строки распределения |
| CASE-07 | Документ 1С проведен по другому договору, связь строки с поставкой подтверждена | `match` | Другой договор допустим: связь с поставкой доказана строкой/распределением | Действие не требуется |
| CASE-08 | Несколько равнозначных кандидатов 1С | `ambiguous_match` или `duplicate_in_1c` | Система не может однозначно выбрать пару | Выполнить ручной разбор дублей |
| CASE-09 | Документ существует в ERP глобально, но не связан с поставкой | `not_linked_to_delivery_in_erp` | Документ существует, принадлежность выбранной поставке не доказана | Проверить связь документа с операцией/поставкой |
| CASE-10 | ERP-сальдо рассчитано, но оборот не подтвержден документами | `balance_status=not_comparable` | Числовая дельта не подтверждает полноту документов | Устранить отсутствующие счета/закрывающие документы |
| CASE-11 | Ставка НДС различается | `vat_mismatch` | Найденная пара имеет разные ставки НДС | Проверить ставку в обоих документах |
| CASE-12 | Документ 1С удален или не проведен | отдельная строка предупреждения/расхождения | Документ существует, но не участвует в учете как проведенный | Проверить состояние документа 1С |
| CASE-13 | Один входящий акт поставщика содержит расходы нескольких поставок | `match` по `document_lines` | В сверке участвуют только строки, распределенные на операции выбранной поставки; сумма шапки акта не сравнивается с частичной суммой поставки | Открыть строки акта и операции ERP при необходимости |

Контрольный пример CASE-13: входящий акт `0ЛБП-000788` на `26 000,00 RUB`. Для поставки ERP `23687` относятся строки `14 798,04` и `3 523,34`, всего `18 321,38`; оставшиеся строки относятся к другой поставке. Клиентская реализация проверяется отдельно: строка `18 321,38` входит в реализацию `0ЛБП-000426`, а услуга `5 937,54` — в реализацию `0ЛБП-000427`, обе по договору покупателя `БП-078924`.

## Протокол пользовательской проверки

Для каждого кейса пользователь отвечает:

1. Что произошло?
2. Кто должен это исправить?
3. Как убедиться, что проблема устранена?

Фиксируются время первичного разбора, переходы в ERP, непонятные термины, дополнительные источники данных и выбранная причина в журнале разбора.
