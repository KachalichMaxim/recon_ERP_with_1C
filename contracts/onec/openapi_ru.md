# 1C REST API для сверки ERP и 1C: документация к OpenAPI

Контракт:

- JSON: `contracts/onec/openapi.json`
- YAML: `contracts/onec/openapi.yaml`

Контракт специально сделан **GET-only**. Python backend ничего не пишет в 1C и не отправляет тело запроса. Все входные данные передаются query-параметрами.

## 1. Что сделать

Нужно реализовать read-only REST API:

```http
GET /reconciliation/v1/health
GET /reconciliation/v1/snapshot
GET /reconciliation/v1/counterparties
GET /reconciliation/v1/contracts
GET /reconciliation/v1/invoices
GET /reconciliation/v1/payments
GET /reconciliation/v1/sales
GET /reconciliation/v1/purchases
GET /reconciliation/v1/account-movements
GET /reconciliation/v1/balances
```

Минимум для первой интеграции:

```http
GET /reconciliation/v1/health
GET /reconciliation/v1/snapshot
```

Остальные GET endpoints нужны для фонового runner-а и массовой сверки.

## 2. Авторизация

Поддержать один из вариантов:

```http
Authorization: Bearer <token>
```

или:

```http
Authorization: Basic <base64(user:password)>
```


## 3. Главный интерактивный запрос

Endpoint:

```http
GET /reconciliation/v1/snapshot
```

Назначение: вернуть один нормализованный снимок 1C по контексту ERP: период, организация, ЮЛ/контрагент, договор, поставка/заявка/спецификация, известные коды документов.

Пример запроса по договору и поставке:

```http
GET /reconciliation/v1/snapshot
  ?date_from=2025-07-01
  &date_to=2025-08-31
  &organization_code=000000001
  &counterparty_inn=7811451960
  &buyer_contract_code=БП-051945
  &committent_contract_code=БП-051946
  &spec_number=921
  &include=contracts
  &include=customer_invoices
  &include=payments
  &include=sales
  &include=purchases
  &include=document_lines
  &include=account_movements
  &include=balances
```

Пример запроса по известным кодам документов 1C:

```http
GET /reconciliation/v1/snapshot
  ?date_from=2025-07-01
  &date_to=2025-08-31
  &organization_code=000000001
  &counterparty_code=БП-000123
  &document_code=ВА-007703
  &document_code=ВА-007704
  &document_code=00БП-010299
  &document_type=customer_invoice
  &document_type=payment
```

### 3.1 Что реально передается в 1C

В 1C передаются **только query-параметры GET-запроса**. Тело запроса отсутствует.

Важно разделять:

| Категория | Где находится | Для чего нужна |
|---|---|---|
| Входные query-параметры | URL `/reconciliation/v1/snapshot?...` | 1C использует их для фильтрации выборки |
| DTO-поля ответа | JSON-массивы `contracts[]`, `payments[]`, `sales[]` и т.д. | 1C возвращает эти поля в ответе, Python использует их для сверки |
| ERP-only поля | остаются в Python/ERP | 1C не получает и не должна искать по ним |

То есть поля из раздела “Маппинг 1C -> DTO” **не передаются в 1C**. Это структура ответа, которую 1C должна вернуть.

### 3.2 Что 1C не получает

1C не получает и не должна использовать как бизнес-фильтр:

| ERP-поле | Почему не передается в 1C |
|---|---|
| `client_id`, `legal_id`, `contact_id` | внутренние id ERP/MariaDB |
| `dog_id` | внутренний id договора ERP |
| `spec_id` | внутренний id поставки/заявки ERP; в 1C не передается, используется только внутри Python как trace |
| `oper_id`, `veda_akts_details_opers.f_id` | внутренние id операций ERP |
| `f_isvozm`, `f_type`, `f_parenttype` | ERP-классификация, 1C ее не знает |
| суммы `get_realizsum`, `get_paidsum` | расчет ERP; 1C возвращает свои документы/движения, Python сравнивает суммы сам |
| номера строк внешних таблиц или тестовых файлов | не production-данные |

Если нужно связать ERP-поставку с 1C, Python передает не `spec_id`, а коды договоров 1C, которые ERP уже получила после выгрузки поставки:

```text
period + organization_code/inn + counterparty_code/inn + buyer_contract_code/committent_contract_code + document_code/document_number
```

Для автоматической сверки документов Python использует ключ `document_type + document_code + document_date + contract_code`.
`document_code` без даты и договора не считается уникальным: в ERP и 1C возможны дубли кодов за разные даты, договоры или типы документов.
Если 1C возвращает несколько документов по одному `document_code + document_date + contract_code`, сервис фиксирует статус `DUPLICATE_IN_1C`, а не выбирает первый документ.

Номер документа нужно разделять на два поля:

| Поле DTO | Что означает |
|---|---|
| `number` | внутренний номер документа 1C, например `00БП-010302` |
| `incoming_number` | входящий номер / номер платежного поручения / номер документа поставщика, например `29191` или `462-722-029396/23/ВА/28/6131` |

Для платежей и поступлений Python не должен сравнивать `incoming_number` с `number`: это разные бизнес-поля. Если `incoming_number` не вернуть, сервис сможет уверенно сравнить код 1C, дату, сумму, валюту и договор, но не сможет проверить входящий номер.

Для поставки основной ключ связи:

| ERP | 1C | Роль |
|---|---|---|
| `veda_specs.f_kod1cb` | `Справочник.ДоговорыКонтрагентов.Код` | договор 1C с покупателем |
| `veda_specs.f_kod1cp` | `Справочник.ДоговорыКонтрагентов.Код` | договор 1C с комитентом / агентский договор |

`contract_number`, `base_contract` и `spec_number` могут передаваться как контрольные/поясняющие признаки, но не должны быть основным способом поиска, если коды `f_kod1cb/f_kod1cp` заполнены.

### 3.3 Минимальные наборы параметров

Для разработчика 1C достаточно поддержать эти рабочие сценарии.

| Сценарий | Что Python передает в 1C | Что 1C должна вернуть |
|---|---|---|
| Сверка договора | `date_from`, `date_to`, `counterparty_inn` или `counterparty_code`, `contract_code` | договоры, счета, оплаты, реализации/акты, движения и остатки по найденному договору |
| Сверка поставки/заявки | `buyer_contract_code` и/или `committent_contract_code`; при наличии `document_code` | документы и строки, связанные с конкретными договорами поставки в 1C |
| Точная сверка документов | `date_from`, `date_to`, повторяемые `document_code`, `document_type` | только найденные документы 1C по указанным кодам/типам |
| Массовая сверка клиента | `date_from`, `date_to`, `counterparty_inn` или `counterparty_code`, `limit`, `cursor` | постраничный набор документов клиента |

### 3.4 Про коды договоров, `base_contract`, `contract_number`, `spec_number`

`buyer_contract_code` - код договора 1C с покупателем. Python берет его из `veda_specs.f_kod1cb`.

`committent_contract_code` - код договора 1C с комитентом / агентского договора. Python берет его из `veda_specs.f_kod1cp`.

Если оба кода заполнены, 1C должна искать документы по точному совпадению:

```text
ДоговорКонтрагента.Код В (&buyer_contract_code, &committent_contract_code)
```

Для строк документов 1C, где договор находится в субконто/аналитике, применяется то же правило: сверяем именно код договора 1C, а не текстовое содержание строки.

Важно: для поставки/заявки этого правила недостаточно. В 1C во вкладке "Документы" могут отображаться связанные документы, у которых прямой `ДоговорКонтрагента.Код` указывает не на договор покупателя/комитента, а на договор поставщика или другой связанный договор. Например, `Документ.ПоступлениеТоваровУслуг` по расходам подрядчика может иметь `contract_code=БП-020846`, но относиться к заявке, где договор покупателя `БП-068417`, а договор комитента `БП-068418`.

Поэтому при запросе по `buyer_contract_code` / `committent_contract_code` 1C должна вернуть:

```text
1. Документы, где ДоговорКонтрагента.Код = buyer_contract_code или committent_contract_code.
2. Документы, которые 1C сама показывает во вкладке "Документы" этой заявки/договора, даже если прямой договор документа другой.
3. Для таких документов заполнить:
   - contract_code = прямой договор документа;
   - linked_contract_codes = коды договоров заявки, из-за которых документ попал в snapshot.
```

Если 1C не может вывести связанные документы заявки через стандартную аналитику, это нужно явно вернуть в `warnings[]`, а не молча отдавать неполный snapshot.

Дата договора-заявки в 1C контролируется отдельно: ожидаемая дата берется из `veda_specs.f_dt`, потому что при выгрузке ERP передает ее в `createSpec` как `specdt`, а 1C записывает это значение в `ДоговорыКонтрагентов.Дата`. Поэтому 1C должна вернуть дату договора в DTO `contracts[]`.

`contract_number` - номер договора, который Python знает из ERP. Для поставки это контрольное поле, а не основной ключ, потому что код договора 1C уже хранится в ERP.

`base_contract` - необязательная подсказка. Она нужна только если в 1C поставки/заявки лежат в иерархии под базовым договором, например:

```text
660/1
  Открытые
    Заявка №921
      договор/строки/акты
```

Если в конфигурации 1C нет понятия базового договора или его нельзя достоверно вывести, 1C может игнорировать `base_contract` как входной фильтр и вернуть `base_contract: null` в DTO.

`spec_number` - бизнес-номер заявки/спецификации, например `921`. Это не `spec_id` ERP. Используется для отображения и дополнительного контроля; основной поиск идет по `buyer_contract_code` / `committent_contract_code`.

### 3.5 Про `include`

`include` - это **не фильтр поиска**, а переключатель состава ответа. Он нужен, чтобы не возвращать лишние тяжелые блоки.

Рекомендуемый минимум для интерактивной сверки:

```http
include=contracts
include=customer_invoices
include=payments
include=sales
include=document_lines
include=account_movements
```

`purchases` и `balances` можно подключать только там, где они нужны конкретному экрану/проверке. Если `include` не передан, 1C может вернуть стандартный набор, но для производительности лучше всегда передавать `include` явно.

## 4. Query-параметры `/snapshot`

| Параметр | Обяз. | Повторяемый | Что передаем | Что делает 1C |
|---|---:|---:|---|---|
| `date_from` | да | нет | начало периода `YYYY-MM-DD` | ограничивает документы по дате |
| `date_to` | да | нет | конец периода `YYYY-MM-DD` | ограничивает документы по дате |
| `request_id` | нет | нет | id запроса Python backend | вернуть в response для трассировки |
| `mode` | нет | нет | `delivery_reconciliation`, `contract_reconciliation`, `client_reconciliation` | режим фильтрации |
| `organization_code` | нет | нет | код организации 1C | фильтр по `Справочник.Организации.Код` |
| `organization_inn` | нет | нет | ИНН организации | резервный поиск организации |
| `organization_name` | нет | нет | название организации | последний резервный поиск |
| `counterparty_code` | нет | нет | код контрагента 1C | фильтр по `Справочник.Контрагенты.Код` |
| `counterparty_inn` | нет | нет | ИНН ЮЛ клиента | фильтр по `Справочник.Контрагенты.ИНН` |
| `counterparty_name` | нет | нет | название ЮЛ клиента | резервный поиск контрагента |
| `buyer_contract_code` | нет | нет | `veda_specs.f_kod1cb` | точный фильтр по договору 1C с покупателем |
| `committent_contract_code` | нет | нет | `veda_specs.f_kod1cp` | точный фильтр по договору 1C с комитентом / агентскому договору |
| `contract_code` | нет | да | коды договоров 1C, включая CB/CP | фильтр по `Справочник.ДоговорыКонтрагентов.Код` |
| `contract_number` | нет | да | номера договоров, например `660/1` | контрольный/резервный фильтр по `ДоговорыКонтрагентов.Номер` |
| `contract_role` | нет | да | `buyer`, `committent`, `supplier`, `agent`, `related` | подсказка роли договора; порядок не является связкой |
| `base_contract` | нет | нет | номер базового договора, например `660/1`; только подсказка | контрольное поле; не заменяет поиск по коду договора 1C |
| `spec_number` | нет | нет | бизнес-номер поставки/заявки/спецификации, например `921` | контрольное поле; не заменяет поиск по коду договора 1C |
| `document_code` | нет | да | коды 1C документов | точный фильтр по коду/номеру 1C |
| `document_number` | нет | да | номера документов | резервный фильтр |
| `document_type` | нет | да | `customer_invoice`, `payment`, `sale`, `purchase`, `closing_document`, `account_movement` | какие типы документов вернуть |
| `include` | нет | да | имена блоков ответа | не фильтрует документы; только ограничивает, какие массивы включить в snapshot |
| `cursor` | нет | нет | курсор страницы | если ответ слишком большой |
| `limit` | нет | нет | размер страницы, default 500, max 2000 | лимит элементов |

Правило минимального фильтра:

```text
date_from + date_to обязательны,
и дополнительно должен быть хотя бы один бизнес-фильтр:
buyer_contract_code / committent_contract_code / contract_code / counterparty_code / counterparty_inn / document_code.
```

## 5. Пагинация

Для `/snapshot` пагинация нужна только если 1C видит, что ответ слишком большой.

Формат:

```http
GET /reconciliation/v1/snapshot?...&limit=500
```

Если есть продолжение, 1C возвращает:

```json
{
  "snapshot": {
    "metadata": {
      "next_cursor": "eyJwYWdlIjoyfQ=="
    }
  }
}
```

Следующий запрос:

```http
GET /reconciliation/v1/snapshot?...&cursor=eyJwYWdlIjoyfQ==
```

Для collection endpoints пагинация обязательна всегда.

Ответ:

{

  "snapshot": {
    "metadata": {
      "source_system": "1c",

Если `include` не передан, 1C возвращает все основные блоки. Если `include` передан, 1C возвращает только указанные блоки.

## 7. Обязательные технические поля DTO

Каждая сущность должна содержать:

| Поле | Что писать |
|---|---|
| `source_system` | всегда `1c` |
| `source_id` | уникальный id объекта 1C, желательно GUID ссылки |
| `source_ref` | код/представление ссылки 1C |
| `source_version` | версия DTO/конфигурации |
| `posted` | документ проведен |
| `deleted` | пометка удаления |
| `status` | `posted`, `draft`, `deleted`, `unposted`, `unknown` |
| `updated_at` | дата/время изменения |
| `hash` | хэш бизнес-полей DTO |

Деньги передавать числом, не строкой. Валюту передавать отдельно в `currency`.

| DTO | 1C |
|---|---|
| `source_id` | `Ссылка.УникальныйИдентификатор()` |
| `source_ref` | `Код` |

| DTO | 1C |
|---|---|

## 9. Как Python будет использовать GET

Python backend строит query так:

```text
date_from/date_to берутся из дат ERP-документов поставки с буфером;
organization_code/inn берутся из ERP ЮЛ организации;
counterparty_code/inn берутся из ERP ЮЛ клиента;
buyer_contract_code берется из veda_specs.f_kod1cb;
committent_contract_code берется из veda_specs.f_kod1cp;
contract_code передается повторяемым списком всех кодов договоров 1C, участвующих в сверке;
contract_number/base_contract/spec_number передаются только как контрольные поля, не как основной ключ;
document_code/document_number передаются только если ERP уже знает код/номер документа 1C.
```

Python **не передает** в 1C `client_id`, `dog_id`, `spec_id`, `oper_id`, id строк ERP и результаты ERP-процедур. Эти значения остаются на стороне Python и используются только после получения DTO от 1C.

Для конкретного договора можно передать только:

```http
GET /reconciliation/v1/snapshot
  ?date_from=2025-01-01
  &date_to=2025-12-31
  &counterparty_inn=7811451960
  &contract_code=БП-051945
  &contract_code=БП-051946
```

Для конкретной поставки лучше передавать:

```http
GET /reconciliation/v1/snapshot
  ?date_from=2025-07-01
  &date_to=2025-08-31
  &counterparty_inn=7811451960
  &buyer_contract_code=БП-051945
  &committent_contract_code=БП-051946
  &spec_number=921
```

## 10. Ошибки

Формат:

```json
{
  "ok": false,
  "error": "validation_error",
  "message": "Не указан date_from",
  "request_id": "spec-20334-20260629123000",
  "details": {}
}
```

Типовые коды:

- `bad_request`;
- `unauthorized`;
- `validation_error`;
- `source_timeout`;
- `internal_error`.

## 11. Что не делать

Не нужно:

- принимать тело запроса;
- публиковать сырые OData-объекты;
- отдавать все реквизиты документов;
- возвращать печатные формы;
- возвращать бинарные вложения;
- создавать или менять документы 1C.
